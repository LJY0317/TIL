### Codex CLI 터미널 전용 사용 가이드
전제:
- 나는 확장 프로그램을 설치할 수 없고, 터미널의 `codex cli`만 사용할 수 있다.
- 2026년 3월 19일 기준 공식 문서상 subagent workflows는 기본 활성화 상태다.
- subagent 활동은 Codex App과 CLI에서 보이고, CLI만으로도 사용할 수 있다.
- 내 로컬 `codex-cli 0.115.0`에서도 `codex features list` 결과 `multi_agent stable true`가 보였다.

핵심 결론:
- 네, 터미널만 써도 `subagents`를 사용할 수 있다.
- 다만 별도의 `codex subagent ...` 같은 전용 명령어를 치는 방식이 아니라, `codex` 세션 하나를 띄운 뒤 그 안에서 "spawn two agents", "use one agent per point", "wait for all"처럼 프롬프트로 지시하는 방식이다.
- 그리고 실행 중인 하위 에이전트는 CLI 내부에서 `/agent`로 확인하고 전환한다.

가장 중요한 오해 방지:
- 내장 `subagents`는 "터미널 2개, 3개를 열어서 각각 `codex`를 따로 실행"하는 방식이 아니다.
- 권장 방식은 "부모 Codex 세션 1개" 안에서 Codex가 하위 에이전트 스레드를 여러 개 띄우는 구조다.
- 같은 저장소, 같은 브랜치에서 OS 터미널 여러 개를 열고 각자 독립 `codex` 세션을 동시에 돌리면, 그건 내장 subagent가 아니라 그냥 별도 세션 여러 개다.
- 이 경우 충돌 위험이 훨씬 커진다.

## 1. 터미널은 몇 개 써야 하나?
### 권장 기본형
터미널 1개만 있으면 된다.

터미널 1:
```bash
cd /path/to/your/repo
codex --cd /path/to/your/repo --search
```

설명:
- `--cd`는 Codex가 작업할 기준 폴더를 지정한다.
- `--search`는 실시간 웹 검색을 켠다. 기본 검색 모드는 `cached`라 최신 정보가 중요하면 `--search`를 켜는 편이 낫다.

### 선택형
터미널 2는 사람이 직접 확인용으로만 쓰는 것이 좋다.

터미널 2:
```bash
cd /path/to/your/repo
git status -sb
```

또는:
```bash
cd /path/to/your/repo
git diff --stat
```

설명:
- 터미널 2는 "관찰용"이다.
- 여기서 또 `codex`를 하나 더 띄우는 것은 권장하지 않는다.

### 비권장
터미널 2, 3, 4에서 각각 같은 저장소/같은 브랜치에 대해 독립 `codex`를 동시에 실행하는 것.

이유:
- 서로 같은 파일을 고칠 수 있다.
- 부모가 자식들을 관리하는 구조가 아니기 때문에 결과 정리가 지저분해진다.
- 공식 문서도 병렬 `write-heavy` 작업은 충돌과 coordination overhead가 커질 수 있다고 설명한다.

- subagent는 CLI 내부 대화와 `/agent`로 다루는 방식이라고 이해하면 된다.

## 4. 정말 중요한 운영 원칙
추천 패턴:
- 메인 에이전트 1명: 요구사항, 판단, 최종 통합
- read-only 서브에이전트 여러 명: 코드 탐색, 테스트 원인 분석, 로그 분석, 문서 확인
- writer 1명: 실제 수정

예외적으로 writer를 2명 이상 써도 되는 경우:
- 수정 파일 집합이 완전히 분리될 때만
- 예: `backend/**`와 `frontend/**`가 명확히 갈라지고, 서로 같은 파일을 건드리지 않을 때

안전한 문장 예시:
```text
Spawn two subagents.
One explores backend only, one explores frontend only.
Do not edit files.
Wait for all, then summarize.
```

```text
Spawn one worker to implement the agreed backend change only.
Do not touch frontend files.
After finishing, summarize the exact files changed and remaining risks.
```

## 5. `AGENTS.md`는 어디에 두나?
Codex는 시작할 때 `AGENTS.md` 계층을 읽는다.

주요 위치:
- 글로벌 기본 규칙: `~/.codex/AGENTS.md`
- 글로벌 임시 덮어쓰기: `~/.codex/AGENTS.override.md`
- 저장소 기본 규칙: `<repo>/AGENTS.md`
- 하위 폴더 전용 임시/강한 규칙: `<repo>/<subdir>/AGENTS.override.md`

작동 방식:
- 글로벌에서는 `AGENTS.override.md`가 있으면 그것만 읽고, 없으면 `AGENTS.md`를 읽는다.
- 프로젝트에서는 루트부터 현재 작업 디렉터리까지 내려오면서 각 디렉터리의 `AGENTS.override.md` 또는 `AGENTS.md`를 읽는다.
- 현재 작업 디렉터리에 가까운 파일일수록 뒤에 합쳐지므로 더 강하게 작동한다.

예시:
```text
~/.codex/AGENTS.md
<repo>/AGENTS.md
<repo>/services/payments/AGENTS.override.md
```

`services/payments`에서 Codex를 실행하면 보통 위 3개가 순서대로 반영된다.

처음 만드는 가장 쉬운 방법:
Codex 세션 안에서
```text
/init
```

설명:
- 현재 디렉터리에 `AGENTS.md` 초안을 만들어준다.
- 만든 뒤 직접 수정해서 프로젝트 규칙에 맞추면 된다.

## 6. `.codex/config.toml`은 뭐고 어디에 두나?
`config.toml`은 Codex 설정 파일이다.

위치:
- 사용자 전체 기본 설정: `~/.codex/config.toml`
- 프로젝트별 설정: `<repo>/.codex/config.toml`

중요:
- 프로젝트의 `.codex/config.toml`은 프로젝트가 trusted일 때만 로드된다.
- 적용이 안 되는 것 같으면 우선 `~/.codex/config.toml`에 넣고, 세션 안에서 `/debug-config`로 확인하는 편이 안전하다.

우선순위:
1. CLI 플래그와 `--config`
2. `--profile`
3. 프로젝트 `.codex/config.toml`
4. `~/.codex/config.toml`
5. 시스템 기본값

## 7. TOML이 뭐냐?
`toml`은 설정 파일 포맷이다.

아주 기본 형태:
```toml
model = "gpt-5.4"
approval_policy = "on-request"
web_search = "live"

[agents]
max_threads = 6
max_depth = 1
```

읽는 법:
- `key = "value"`: 단일 설정
- `[agents]`: 섹션 시작
- 섹션 아래의 키들은 그 그룹의 설정

실무적으로 자주 쓰는 키:
- `model`
- `model_reasoning_effort`
- `approval_policy`
- `sandbox_mode`
- `web_search`
- `[agents].max_threads`
- `[agents].max_depth`

## 8. 바로 복붙 가능한 최소 설정 예시
### `~/.codex/config.toml`
```toml
model = "gpt-5.4"
model_reasoning_effort = "medium"
approval_policy = "on-request"
web_search = "cached"
personality = "pragmatic"

[agents]
max_threads = 6
max_depth = 1

[profiles.review]
model = "gpt-5.4"
model_reasoning_effort = "high"
approval_policy = "on-request"

[profiles.fast-scan]
model = "gpt-5.4-mini"
model_reasoning_effort = "medium"
approval_policy = "on-request"
```

사용 예시:
```bash
codex --profile review --cd /path/to/repo
codex --profile fast-scan --cd /path/to/repo
```

### `<repo>/AGENTS.md`
```md
# Repository Working Rules

- 변경 전 관련 파일과 테스트 경로를 먼저 파악할 것
- 구현 전에 영향 범위를 요약할 것
- 수정 후 관련 테스트/검증 명령을 제안할 것
- 같은 요청에서 불필요한 대규모 리팩터링은 하지 말 것
```

## 9. `.codex/agents/*.toml`은 뭐고 어떻게 쓰나?
이건 "커스텀 에이전트 정의 파일"이다.

둘 수 있는 위치:
- 개인용: `~/.codex/agents/*.toml`
- 프로젝트용: `<repo>/.codex/agents/*.toml`

공식 문서 기준 필수 키:
- `name`
- `description`
- `developer_instructions`

선택 키:
- `model`
- `model_reasoning_effort`
- `sandbox_mode`
- `nickname_candidates`

Codex에는 기본 내장 agent도 있다:
- `default`
- `worker`
- `explorer`

즉:
- 커스텀 에이전트를 안 만들어도 subagent 사용은 가능하다.
- 커스텀 파일을 만들면 특정 역할을 더 정밀하게 고정할 수 있다.

### 예시 1: 읽기 전용 탐색 에이전트
파일: `<repo>/.codex/agents/pr-explorer.toml`
```toml
name = "pr_explorer"
description = "Read-only code explorer for tracing affected code paths."
model = "gpt-5.4-mini"
model_reasoning_effort = "medium"
sandbox_mode = "read-only"
developer_instructions = """
Stay in exploration mode.
Trace the real execution path.
Report files, symbols, and likely impact.
Do not edit files.
"""
```

### 예시 2: 구현 전용 에이전트
파일: `<repo>/.codex/agents/implementer.toml`
```toml
name = "implementer"
description = "Implementation-focused worker for small targeted changes."
model = "gpt-5.4"
model_reasoning_effort = "medium"
developer_instructions = """
Implement only the agreed change set.
Avoid unrelated edits.
Summarize exact files changed and remaining risks.
"""
```

사용 예시:
```text
Use pr_explorer and implementer in this workflow.
First let pr_explorer inspect the affected paths in read-only mode.
Wait for the result.
Then let implementer apply only the agreed minimal fix.
```

## 10. 워크스페이스 폴더 지정은 어떻게 하나?
가장 기본:
```bash
codex --cd /path/to/repo
```

추가로 다른 폴더까지 쓰기 허용이 필요하면:
```bash
codex --cd /path/to/repo --add-dir /path/to/shared-lib
```

설명:
- `--cd`: 메인 작업 루트
- `--add-dir`: 추가 writable root

프로젝트 루트 판단:
- 기본적으로 `.git`이 있는 디렉터리를 프로젝트 루트로 본다.
- 필요하면 `project_root_markers`로 바꿀 수 있다.

예시:
```toml
project_root_markers = [".git", ".hg", ".sl"]
```

## 11. 실전 추천 워크플로우
### A. 코드 리뷰/조사
터미널 1:
```bash
cd /path/to/repo
codex --profile review --cd /path/to/repo --search
```

프롬프트:
```text
Review this branch with parallel subagents.
Spawn one subagent for security risks, one for test gaps, and one for maintainability.
Wait for all three, then summarize the findings by category with file references.
Do not edit files.
```

### B. 구현 전 탐색
프롬프트:
```text
Spawn two explorer-style subagents.
One traces the backend execution path.
One traces the frontend execution path.
Both must stay read-only.
Wait for all, then propose a minimal implementation plan.
```

### C. 구현
프롬프트:
```text
Now implement the agreed plan.
Use only one writer unless the file ownership is clearly disjoint.
After changes, summarize exact files changed and run a review pass.
```

## 12. 별도 OS 터미널 여러 개를 꼭 쓰고 싶다면
같은 워킹트리에서 하면 안 된다.

대신 `git worktree`로 분리하는 것이 맞다.

예시:
```bash
cd /path/to/repo
git worktree add ../repo-backend -b codex-backend HEAD
git worktree add ../repo-frontend -b codex-frontend HEAD
```

그 다음:

터미널 1:
```bash
codex --cd ../repo-backend
```

터미널 2:
```bash
codex --cd ../repo-frontend
```

이 방식은:
- 파일 충돌 가능성을 줄이고
- 각 세션의 책임 범위를 나누고
- 마지막에 git 기준으로 병합하기 쉽다

하지만 이건 "내장 subagents"가 아니라 "독립 Codex 세션 여러 개"이므로, 가능하면 먼저 내장 subagent 방식을 우선 사용하는 것이 좋다.

## 13. 기억해둘 디렉터리/파일
- `~/.codex/AGENTS.md`: 내 모든 프로젝트에 공통으로 먹는 규칙
- `~/.codex/AGENTS.override.md`: 임시 글로벌 덮어쓰기
- `~/.codex/config.toml`: 내 기본 설정
- `~/.codex/agents/*.toml`: 개인 커스텀 agent
- `<repo>/AGENTS.md`: 저장소 공통 규칙
- `<repo>/<subdir>/AGENTS.override.md`: 특정 하위 폴더 전용 강한 규칙
- `<repo>/.codex/config.toml`: 프로젝트별 설정
- `<repo>/.codex/agents/*.toml`: 프로젝트 전용 커스텀 agent

## 14. 나한테 맞는 가장 현실적인 추천
처음에는 이렇게만 쓰는 게 제일 좋다:

1. `~/.codex/config.toml` 하나 만든다.
2. 프로젝트 루트에 `AGENTS.md` 하나 만든다.
3. 터미널 1개에서 `codex --cd <repo> --search`로 시작한다.
4. 먼저 read-only subagent 2~3명으로 탐색시킨다.
5. 구현은 writer 1명만 시킨다.
6. 마지막에 `/review`로 한 번 더 검토한다.

즉, "멀티 에이전트 = 터미널 여러 개"라고 생각하지 말고,
"부모 세션 1개 안에서 read-heavy 병렬 조사 + write는 최소화"라고 생각하는 것이 가장 안전하고 효율적이다.

## 15. AgriBot 현재 상태 기준 더 효율적인 3분할
현재 저장소는 `frontend` 코드가 아직 없고, 반대로 `agribot_description`, `agribot_navigation`, `agribot_bringup`, `backend`는 이미 어느 정도 뼈대가 있다.

그래서 지금 단계에서는 "agribot / backend / frontend"로 나누는 것보다 아래처럼 나누는 편이 더 효율적이다.

- Agent 1: ROS 코어/시뮬레이션 축
  - 대상: `agribot_description`, `agribot_navigation`, `agribot_bringup`, `agribot_interfaces`
  - 이유: 실제로 이미 동작 자산이 있고, TF/topic/map/waypoint/launch 흐름이 한 덩어리다.
- Agent 2: 미구현 런타임 축
  - 대상: `agribot_perception`, `agribot_control`, `agribot_iot`
  - 이유: 지금 프로젝트의 가장 큰 공백이자, P0 완주를 막는 핵심 연결고리다.
- Agent 3: 서비스 계약/클라이언트 준비 축
  - 대상: `backend`, `API_명세서.md`, `DB_SCHEMA_초안.sql`, 기획서
  - 이유: 프론트엔드 코드는 아직 없으므로, 지금은 UI 자체보다 API/DB/MQTT/WebSocket 계약을 고정하는 것이 먼저다.

즉, 현재는 "화면별 분업"보다 "의존성과 리스크 기준 분업"이 더 맞다.

프롬프트:
```text
현재 AgriBot 저장소를 읽기 전용으로 진단해.
코드는 수정하지 말고, 병렬 subagent 3명을 쓰되 현재 상태에 맞게 아래처럼 나눠.

1. Explorer A: ROS 코어/시뮬레이션 축
- 대상: /home/ssafy/SSAFY/S14P21A602/agribot_ws/src/agribot_description
- 대상: /home/ssafy/SSAFY/S14P21A602/agribot_ws/src/agribot_navigation
- 대상: /home/ssafy/SSAFY/S14P21A602/agribot_ws/src/agribot_bringup
- 대상: /home/ssafy/SSAFY/S14P21A602/agribot_ws/src/agribot_interfaces
- 해야 할 일:
  현재 launch 흐름, TF/topic 연결, map/waypoint/config/test 자산, 재사용 가능한 기존 구현을 정리해.
  이미 안정적인 부분과 P0 데모를 막는 부분을 구분해.

2. Explorer B: 미구현 런타임 축
- 대상: /home/ssafy/SSAFY/S14P21A602/agribot_ws/src/agribot_perception
- 대상: /home/ssafy/SSAFY/S14P21A602/agribot_ws/src/agribot_control
- 대상: /home/ssafy/SSAFY/S14P21A602/agribot_ws/src/agribot_iot
- 해야 할 일:
  perception -> mission/control -> iot로 이어지는 런타임 체인에서 무엇이 비어 있는지 정리해.
  필요한 노드, msg/srv/action 확장, 가장 작은 구현 단위를 제안해.
  P0 데모를 완성하는 최소 연결 순서를 제시해.

3. Explorer C: 서비스 계약/클라이언트 준비 축
```

## 16. Agribot 자율 매핑 리팩토링용 최종 프롬프트 초안
인터넷으로 공식 문서와 대표 라이브러리를 다시 확인한 뒤 정리한 메모:

- `Nav2`는 강력한 범용 내비게이션 프레임워크지만, "미지 공간 자동 탐사" 자체를 완성품으로 제공하는 스택은 아니다.
- `m-explore-ros2 / explore_lite`는 존재하는 커뮤니티 기반 frontier exploration 패키지이지만, 공식 Nav2 구성요소가 아니고, 이것만 붙인다고 로보락 수준의 완주성이 보장되지는 않는다.
- `slam_toolbox`는 LiDAR와 odometry를 이용해 pose graph를 만들고 `map -> odom` 변환과 지도를 publish한다.
- `robot_localization`의 `ekf_localization_node` / `ukf_localization_node`는 IMU + wheel odometry 융합에 적합하고, 2D 로봇이면 `two_d_mode`가 핵심이다. 다만 EKF는 "특징점이 거의 없는 넓은 공간에서 영원히 localization을 잃지 않게 해주는 마법"은 아니다.
- `Nav2`에는 `BackUp`, `Spin`, `DriveOnHeading`, `Collision Monitor`, `Keepout Filter`, `Coverage Server(opennav_coverage)` 같은 검증된 기능이 이미 있다.
- `opennav_coverage` / `Fields2Cover`는 "알려진 field/zone/rows"의 complete coverage에는 매우 유용하지만, "아직 지도가 없는 미지 공간 매핑"을 대신하는 frontier 탐색기는 아니다.
- 따라서 Agribot에 필요한 것은 "Nav2만 믿는 단일 알고리즘"이 아니라, `센서 융합 + 안정적인 SLAM + 탐색 supervisor + recovery state machine + 필요시 coverage`를 합친 하이브리드 구조다.
- 로보락의 핵심 아이디어는 그대로 복붙할 수 있는 공개 라이브러리가 아니라, `직선 우선`, `벽/경계 먼저 확보`, `상태 머신 기반 모드 전환`, `끊기면 확실히 빠져나오는 recovery`, `센서 융합`의 조합으로 보는 것이 맞다.

아래 프롬프트를 다음 구현 요청에 사용:

```text
현재 ROS 2 Jazzy + Gazebo Harmonic 기반 Agribot 저장소의 자율주행 매핑 구조를 전면 재설계해줘.

중요:
- "/home/ssafy/SSAFY/S14P21A602/docs/수확해조_프로젝트_개발_계획_및_협업_가이드.md"를 반드시 참고해.
- 현재 프로젝트는 LiDAR, IMU, diff-drive 기반이고, Nav2 / slam_toolbox / Gazebo를 사용한다.
- 목표는 "로보락 같은 상용 청소기 수준의 방향성"을 참고하되, 과장된 100% 완벽 보장은 주장하지 말고, 현재 Agribot의 하드웨어/시뮬레이션 조건에서 가장 현실적이고 재현 가능한 구조로 리팩토링하는 것이다.
- 단순 frontier 탐색기 하나로 끝내지 말고, unknown-space mapping 문제와 known-map coverage 문제를 분리해서 설계해.
- 기존 코드베이스를 실제로 읽고, 현재 토픽/TF/launch/odom/scan/costmap/behavior tree 상태를 확인한 뒤 작업해.
- 구현 후에는 테스트, 검증 명령, 한계, 남은 리스크까지 정리해.

내가 원하는 것은 "그냥 잘 되게 해줘"가 아니라 아래 아키텍처를 실제 코드로 설계/구현하는 것이다.

1. 현실 판단
- 먼저 현재 Agribot의 문제를 정확히 진단해.
- 증상 예시:
  - 시작점에서 좌우로 씰룩거리며 앞으로 못 나감
  - 좁은 곳에 들어가거나 들어가려다 끼어서 제자리 회전 루프 발생
  - 넓은 공간에서 다음 goal을 못 잡고 같은 자리에서 빙글빙글 돎
  - TF / odom / slam time mismatch 때문에 초반 bringup이 불안정함
- 원인을 다음 축으로 나눠서 분석해:
  - 상태 추정(odometry, IMU, EKF, TF)
  - SLAM(slam_toolbox)
  - Nav2 planner/controller/behavior tree
  - frontier 또는 coverage supervisor
  - Gazebo / sensor bridge / use_sim_time

2. 반드시 반영할 외부 전략
- 공식/대표 오픈소스 전략을 참고하되, 맹목적으로 한 라이브러리에 올인하지 마.
- 아래를 비교하고, 현재 Agribot에 맞는 선택을 명시적으로 해:
  - Nav2 + custom frontier supervisor
  - m-explore-ros2(explore_lite ROS2 port) 사용 여부
  - robot_localization EKF/UKF 도입 여부
  - Nav2 Regulated Pure Pursuit vs MPPI controller 선택
  - Collision Monitor / Keepout Filter 사용 여부
  - opennav_coverage / Fields2Cover를 "매핑 완료 후 coverage" 단계에만 사용할지 여부
- 특히 다음 점을 반영해:
  - `explore_lite`는 쓸 수 있지만 만능 해결책은 아니다
  - `opennav_coverage`는 known field complete coverage용이지, unknown map frontier exploration 대체제가 아니다
  - `robot_localization`은 odom/IMU를 안정화하지만 featureless 환경에서 localization 문제를 완전히 없애지는 못한다

3. 목표 아키텍처
아래 구조를 기본안으로 삼아 구현해줘.

3-1. 상태 추정 계층
- `robot_localization` EKF를 추가해서 wheel odometry + IMU를 융합해.
- diff-drive 2D 로봇 기준 `two_d_mode`를 올바르게 사용해.
- `map`, `odom`, `base_link` 프레임 역할을 명확히 정리해.
- `slam_toolbox`가 `map -> odom`을 담당할 때와 EKF가 `odom -> base_link`를 담당할 때 TF 충돌이 없게 해.
- 넓은 공간에서 직진 시 yaw drift가 얼마나 줄어드는지 확인 가능한 설정/검증 절차를 포함해.

3-2. 매핑 계층
- `slam_toolbox`를 기준으로 mapping mode를 안정화해.
- 실제 LiDAR topic, odom topic, transform tolerance, use_sim_time을 현재 Agribot launch 구조와 맞춰 정리해.
- mapping 중 `map`이 멈추거나 뒤틀리는 원인을 줄이기 위한 필수 파라미터와 로그 확인 포인트를 정리해.

3-3. 주행 제어 계층
- Nav2 controller는 기본적으로 `Regulated Pure Pursuit`를 우선 검토해.
- 이유:
  - diff-drive에서 직진 우선 성향을 만들기 쉽고
  - 비교적 계산량이 안정적이며
  - industrial/service robot용으로 잘 정리되어 있다
- 다만 현재 환경에서 RPP 한계가 명확하면 MPPI를 대안으로 비교해도 된다.
- DWB를 쓴다면 왜 RPP/MPPI 대신 DWB를 유지하는지 분명히 설명해.
- 목표는 "로봇청소기처럼 직선 구간을 최대한 길고 안정적으로 주행"하는 것이지, 작은 heading 차이에 계속 출렁이는 것이 아니다.

3-4. 탐색 supervisor 계층
Nav2 자체 goal-to-goal 내비게이션 위에, 별도의 상위 탐색 supervisor 노드를 설계/구현해.

이 supervisor는 최소한 아래 상태를 가져야 한다.

- `BOOTSTRAP_FIND_WALL`
  - 시작 직후 frontier가 불안정하거나 주변이 너무 텅 비었을 때
  - EKF heading을 믿고 `DriveOnHeading` 중심으로 먼저 직진한다
  - 앞쪽 LiDAR에 장애물이 일정 거리 이하로 들어오면 정지한다
  - 최대 전진 거리 또는 timeout도 둔다

- `BOUNDARY_FOLLOW`
  - 처음 벽이나 경계를 찾았으면 일정 거리 offset을 유지하며 외곽을 따라 돈다
  - 목적은 방/비닐하우스의 외곽선을 먼저 안정적으로 잡고, slam_toolbox가 map skeleton을 만들도록 돕는 것이다
  - 필요하면 간단한 wall-follow PID 또는 heading 보정 로직을 직접 구현한다

- `FRONTIER_EXPLORE`
  - 외곽이 어느 정도 생긴 뒤에는 frontier 기반으로 미탐색 공간을 메운다
  - 기존 custom frontier explorer를 개선할지, m-explore-ros2를 붙일지, 둘을 혼합할지 선택해
  - frontier goal은 단순 최근접이 아니라 아래 요소를 점수화해:
    - distance
    - cluster size
    - local free-space support
    - heading continuity
    - stuck history / blacklist

- `RECOVERY_ESCAPE`
  - stuck 판단 시 무의미한 spin loop를 금지해
  - recovery 우선순위는 예를 들어 아래처럼 구성해:
    1. BackUp
    2. 180도 Spin 또는 open heading 방향 회전
    3. DriveOnHeading으로 짧고 강한 탈출 직진
    4. goal blacklist 후 frontier 재선정
  - recovery는 Behavior Tree와 supervisor 둘 다에서 중복/충돌 없이 동작해야 한다

- `COVERAGE_FILL`
  - 이 단계는 "unknown map을 만드는 단계"가 아니라, 외곽과 자유공간이 어느 정도 확보된 뒤 coverage 품질을 높이는 선택적 단계다
  - 맵이 안정화되면 `opennav_coverage` / `Fields2Cover` 또는 유사 boustrophedon 전략을 연결해 "내부를 직선 기반으로 채우는" 것을 검토해
  - 단, 이 단계는 매핑 초기 bootstrap보다 뒤에 와야 한다

4. 로보락식 아이디어를 Agribot에 맞게 재해석
로보락 알고리즘을 그대로 복붙할 수는 없으므로, 아래 원칙을 오픈소스 스택 위에 재구성해줘.

- 직선 우선
  - heading이 크게 어긋나지 않으면 먼저 전진한다
  - 사소한 heading 오차로 좌우로 흔들리지 않도록 controller / progress / goal tolerance를 재튜닝한다

- 경계 먼저 확보
  - 초기 허허벌판에서는 무작정 frontier를 찾으려 하지 말고, 먼저 직진해서 벽/경계를 찾고 경계를 따라 map outline을 만든다

- 모드 전환
  - "Find Wall -> Boundary Follow -> Frontier Explore -> Coverage Fill" 같은 state machine을 명시적으로 구현한다
  - 지금처럼 모든 문제를 Nav2 planner/controller/recovery 하나에 떠넘기지 마

- 확실한 탈출
  - 좁은 통로 stuck에서는 spin보다 `BackUp + TurnAround + DriveOut`을 우선한다

- 센서 융합
  - 직진 구간에서 LiDAR feature가 적을수록 IMU + odom의 역할이 더 커지므로 EKF 세팅을 꼭 포함한다

5. Nav2 튜닝 방침
- costmap inflation, footprint padding, obstacle layer range, progress checker, goal checker, transform_tolerance을 현재 Agribot 크기와 비닐하우스 폭에 맞춰 다시 설계해
- 시작점에서 "겁먹고 제자리 도는" 현상을 줄이는 방향으로 조정해
- recovery BT에서 `Spin`, `BackUp`, `DriveOnHeading`를 실제로 활용해
- 필요하면 `Collision Monitor`를 controller 아래 safety filter로 넣어서, 좁은 곳에서 과도한 돌진이나 진동을 줄여
- 필요하면 `Keepout Filter` 또는 boundary mask를 이용해 절대 들어가면 안 되는 구역 / 반복해서 stuck 나는 구역을 막아
- planner는 partial planning, replanning 주기, unknown 처리 방식까지 재검토해
```

### Codex 세션 재개와 subagent 재사용 메모

핵심 결론:
- 매번 긴 프롬프트를 처음부터 다시 입력할 필요는 없다.
- 터미널이나 컴퓨터를 껐다 켜도 `~/.codex` 아래에 세션 기록이 남아 있으면, 이전 Codex 세션을 다시 열 수 있다.
- 다만 종료 전 돌고 있던 "활성 subagent 작업"이 그대로 살아있다고 가정하면 안 된다.
- 가장 안전한 흐름은 `부모 세션 resume -> 필요하면 subagent 다시 spawn` 이다.

확인한 명령어:
```bash
codex resume --last
codex resume <SESSION_ID>
codex fork --last
codex fork <SESSION_ID>
```

의미:
- `resume`: 예전 세션을 그대로 이어서 작업한다.
- `fork`: 예전 세션의 맥락은 가져오되, 새 갈래 세션으로 시작한다.

실전 사용 방법:
1. 먼저 가장 최근 세션을 다시 연다.

```bash
codex resume --last
```

2. 세션이 열리면 예전 대화 맥락을 바탕으로 바로 이어서 요청한다.
3. 만약 이전에 쓰던 subagent들이 이미 끝났거나 사라졌다면, 그때만 다시 spawn한다.

중요한 구분:
- 재개 가능한 것: 부모 Codex 세션의 대화/문맥
- 그대로 유지된다고 믿으면 안 되는 것: 종료 전 실행 중이던 하위 에이전트의 live 상태

따라서 실무적으로는:
- 긴 설명 프롬프트를 매번 새로 칠 필요는 없다.
- 먼저 `codex resume --last`를 시도한다.
- 이어서 "이전 작업 흐름 그대로 계속하고, 필요하면 subagent만 다시 띄워줘" 식으로 요청하면 된다.

재스폰 프롬프트를 매번 안 치는 방법:
- 자주 쓰는 subagent 스폰 프롬프트를 파일로 저장해두면 된다.
- 예: `prompts/subagent_spawn.txt`, `prompts/review_parallel.txt`
- 그러면 세션을 못 살렸을 때도 내용을 다시 타이핑하지 않고 파일을 열어 붙여 넣거나, 그 내용을 그대로 다시 요청하면 된다.

예시로 저장해둘 짧은 템플릿:
```text
이 작업을 병렬 subagent workflow로 처리해줘.

규칙:
- 먼저 전체 작업을 짧게 분석
- 바로 막히는 핵심 작업은 부모가 직접 처리
- 독립적인 조사, 테스트, 파일별 구현은 subagent로 분리
- 서로 같은 파일을 동시에 수정하지 않게 역할을 나눌 것
- 각 subagent 결과는 변경 파일, 핵심 판단, 검증 결과만 짧게 보고
```

정리:
- 세션 복구가 되면 `resume`
- 같은 맥락으로 새 분기면 `fork`
- subagent는 필요 시 재생성
- 자주 쓰는 프롬프트는 파일로 저장해서 재입력 시간을 줄이기

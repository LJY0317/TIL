- backend도 문서와 구현 차이가 크다. API 응답 형식, DB 스키마 방향, 필드명, ID 전략이 아직 고정되지 않았다.
- 특히 DB 접속 정보 하드코딩, `create_all` 기반 자동 테이블 생성, placeholder route 위주 구조는 팀 협업 시 동일 결과 재현성을 해칠 가능성이 크다.

### 왜 RViz 맵이 로보락처럼 깔끔하지 않고 지저분해 보이는가

지금 보이는 화면은 "사용자용으로 다듬어진 최종 맵"이 아니라, 개발 중 디버깅을 위해 raw 정보가 많이 겹쳐진 RViz 화면에 가깝다.

- RViz 설정상 `/map`만 보이는 것이 아니라 `LaserScan`, `Global Costmap`, `Local Costmap`도 함께 켜져 있다.
  - 그래서 흰 점 형태의 raw scan, costmap의 보라/하늘색 영역, SLAM occupancy map이 한 화면에 겹쳐 보여 더 복잡하게 보인다.
- 현재 온실 월드는 벽만 라이다에 잡히는 구조가 아니다.
  - `tomato_plant/model.sdf`에 식물 줄기/덩굴 덩어리를 따라가는 collision box를 넣어, 2D LiDAR가 작물도 지속적인 장애물로 인식하도록 해두었다.
  - 즉 작물 자체가 맵에 남는 것이 의도된 구조다.
- `farm_world.sdf`에도 온실 외벽과 재배 베드를 단순 collision proxy로 따로 넣어두었다.
  - 그래서 실내 청소기처럼 벽 위주의 단순 평면도가 아니라, 통로 + 베드 + 작물 구조가 함께 반영된다.
- LiDAR 설정도 비교적 공격적이다.
  - 현재 GPU LiDAR는 `720 samples`, `25.0m range`로 설정되어 있어 먼 구조물과 주변 장애물을 많이 주워온다.
  - 이는 "예쁜 지도"보다 "실제 주행 판단에 필요한 장애물 보존" 쪽에 더 가깝다.
- SLAM Toolbox 쪽에도 로보락 같은 consumer용 map beautification 단계는 없다.
  - 현재는 `0.05m` 해상도의 raw occupancy map을 실시간으로 쌓는 구조라, 벽 직선화/노이즈 제거/방 분할 같은 후처리가 자동으로 들어가지 않는다.
- 추가로 문서화된 현재 제한 사항대로 `farm_world.sdf`의 Gazebo 물리 부하 때문에 `odom -> base_link` TF가 늦거나 흔들릴 수 있다.
  - 이런 오도메트리/TF 흔들림은 scan 누적 결과를 퍼지게 만들어 맵 가장자리를 더 지저분하게 보이게 할 수 있다.

정리하면, 지금 맵이 지저분해 보이는 것은 LiDAR가 이상해서라기보다
"raw scan + live SLAM map + Nav2 costmap + 작물 장애물 반영 + TF 흔들림 가능성"이 함께 보이는 개발용 화면이기 때문이다.

반대로 로보락 앱에 보이는 맵은 보통 아래 후처리를 거친 사용자용 결과물에 가깝다.

- 벽 직선화
- 작은 노이즈 제거
- 일시 장애물 억제
- 방 단위 분할
- UI용 단순화 렌더링

즉 현재 AgriBot 맵은 "깔끔한 집 평면도"를 우선한 것이 아니라, 온실 자율주행에 필요한 실제 장애물 정보를 더 많이 살리는 방향으로 만들어져 있다고 보는 편이 맞다.

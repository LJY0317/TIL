## [S14P-102] [BE] 로봇 관제용 map·pose·semantic layer API 구축

### 개요

S14P-102 로봇 관제 화면 연동을 위한 backend API 정리를 진행했습니다.

이번 MR의 목적은 다음 3가지입니다.

1. 프론트 `/robot` 화면이 실제 정적 맵과 현재 위치를 읽을 수 있도록 `map`, `pose`, `status` API를 정리한다.
2. 식물, 스프링클러, row guide 같은 semantic layer를 backend 계약으로 제공한다.
3. DB가 아직 완전히 준비되지 않은 환경에서도 로봇 관제 API와 CORS가 먼저 동작하도록 보호한다.

이번 작업으로 backend는 정적 map 메타데이터, raw PGM 경로, live/fallback pose, semantic layer, 구조화된 robot status를 프론트가 바로 붙일 수 있는 형태로 반환할 수 있게 정리됐습니다.

---

### 배경

S14P21A602-101까지 진행한 시점에는 프론트가 `/robot` 지도 화면용 semantic layer UI를 먼저 준비한 상태였습니다.

하지만 backend 쪽에는 아래와 같은 빈 부분이 있었습니다.

- `/api/v1/robot/map`, `/api/v1/robot/pose`, `/api/v1/robot/map/layers` 같은 실제 관제 계약이 없음
- `backend/routers/robots.py`의 `/status`, `/pose`가 placeholder 중심이라 현재 위치와 정적 맵 정보를 프론트가 신뢰하고 쓰기 어려움
- DB가 아직 안 떠 있는 개발 환경에서는 backend 시작 자체가 깨질 수 있음
- 프론트 개발 서버 `localhost:5173`에서 직접 API를 호출할 CORS 설정이 없음

즉, 프론트가 먼저 완성한 지도 UI를 실제 데이터와 붙일 수 있도록 **backend 쪽 API 기준선**을 먼저 만드는 작업이 필요했습니다.

---

### 주요 변경 사항

### 1. robot map 서비스 계층 분리

- `backend/robot_map_service.py`를 추가해 라우터 밖으로 파싱 로직을 분리했습니다.
- 이 서비스 계층은 다음 자산을 읽어 API 응답용 payload를 만듭니다.
- `farm_map.yaml`
- `farm_map.pgm`
- `crop_instances.yaml`
- `farm_world.sdf`
- `iot_devices.yaml`
- `/tmp/agribot_runtime/robot_pose_snapshot.json`

또한 snapshot이 없거나 오래된 경우에도 프론트가 화면을 유지할 수 있도록 fallback pose를 반환하도록 구성했습니다.

### 2. `/api/v1/robot/*` 관제 API 추가 및 구조화

- `backend/routers/robots.py`를 정리해 아래 계약을 제공합니다.
- `GET /api/v1/robot/map`
- `GET /api/v1/robot/map/raw`
- `GET /api/v1/robot/map/layers`
- `GET /api/v1/robot/pose`
- `GET /api/v1/robot/status`

모든 응답은 프론트가 바로 붙이기 쉽게 `data` 래퍼와 `source: live/fallback` 기준을 포함하도록 통일했습니다.

### 3. semantic layer를 backend 계약으로 정리

- `crop_instances.yaml`에서 plant / tomato 정보를 읽습니다.
- `farm_world.sdf`에서 sprinkler와 row guide를 읽습니다.
- `iot_devices.yaml`에서 logical device 정보를 읽습니다.

이렇게 해서 프론트 fallback 하드코딩이 아니라 backend 계약 기반으로 semantic layer를 붙일 수 있는 기준선을 만들었습니다.

### 4. 개발 환경 보호 및 CORS 추가

- `backend/main.py`에서 `localhost:5173`, `127.0.0.1:5173` CORS를 허용했습니다.
- DB 초기화 실패를 비치명적으로 처리해, DB가 없어도 로봇 지도 API는 먼저 뜰 수 있도록 보호했습니다.

## [S14P-103] [ROS] 로봇 관제 연동용 pose 및 semantic snapshot exporter 구축

### 개요

S14P-103 로봇 관제 연동용 runtime snapshot export를 진행했습니다.

이번 MR의 목적은 다음 3가지입니다.

1. Gazebo/Nav2 실행 중인 로봇의 현재 위치를 backend가 읽을 수 있는 JSON snapshot으로 저장한다.
2. 식물, 스프링클러, 논리 장치 연결 정보를 semantic layer snapshot으로 함께 정리한다.
3. 기존 `simulation.launch.py` 실행 명령은 유지한 채, 팀원 누구나 같은 런치로 같은 산출물을 얻을 수 있게 한다.

이번 작업으로 `map -> base_link` TF가 있을 때는 해당 pose를, 아직 없을 때는 odometry fallback pose를 JSON으로 기록할 수 있게 정리했고, 현재 world/config 기준 식물 및 설비 레이어도 backend가 바로 읽을 수 있는 형태로 함께 남기도록 구성했습니다.

---

### 배경

S14P21A602-102까지 진행한 시점에는 backend가 `/api/v1/robot/pose`, `/api/v1/robot/map/layers` 같은 API를 통해 프론트에 로봇 위치와 semantic layer를 내려주려는 구조가 먼저 준비되어 있었습니다.

하지만 `agribot_ws` 쪽에는 아래와 같은 빈 부분이 있었습니다.

- Gazebo/Nav2가 실행 중이어도 backend가 쉽게 읽을 수 있는 pose snapshot 파일이 없음
- `crop_instances.yaml`, `farm_world.sdf`, `iot_devices.yaml`에 흩어진 정보를 runtime 기준 JSON 계약으로 묶어 주는 계층이 없음
- 사용자는 기존 launch 명령을 그대로 쓰길 원하지만, 별도 export 노드가 아직 연결되어 있지 않음

즉, ROS 내부 상태와 backend REST API 사이를 이어 주는 **가벼운 파일 기반 export 계층**이 필요했습니다.

---

### 주요 변경 사항

### 1. runtime snapshot 서비스 계층 추가

- `agribot_bringup/runtime_snapshot_service.py`를 추가했습니다.
- 이 파일은 아래 자산을 읽어 backend 친화적인 JSON 계약으로 정리합니다.
- `agribot_navigation/maps/farm_map.yaml`
- `agribot_navigation/maps/farm_map.pgm`
- `agribot_description/config/crop_instances.yaml`
- `agribot_description/worlds/farm_world.sdf`
- `agribot_iot/config/iot_devices.yaml`

또한 패키지 install share 경로를 우선 사용하고, 개발 중에는 source tree fallback도 지원하도록 구성해 팀원 환경 차이에도 견딜 수 있게 했습니다.

### 2. pose snapshot exporter 노드 추가

- `agribot_bringup/runtime_snapshot_exporter.py`를 새로 추가했습니다.
- 이 노드는 주기적으로 현재 로봇 위치를 읽어 `/tmp/agribot_runtime/robot_pose_snapshot.json`에 저장합니다.
- pose 계산 우선순위는 다음과 같습니다.
- `map -> base_link` TF가 있으면 그 pose 사용
- TF가 아직 없으면 `/odometry/filtered`
- 그것도 없으면 `/odom`

snapshot에는 아래 정보가 포함됩니다.

- `x`
- `y`
- `yaw`
- `frame_id`
- `timestamp`
- `source_mode`

### 3. semantic layer snapshot 함께 저장

- 같은 exporter 흐름에서 `/tmp/agribot_runtime/robot_map_layers_snapshot.json`도 함께 저장하도록 구성했습니다.
- 현재는 다음 정보를 포함합니다.
- 식물 24주 위치
- 스프링클러 4개 위치
- 논리 장치 연결 정보
- 지도 메타데이터

즉, backend는 이후 world/config를 직접 다시 파싱하지 않고도 이 snapshot을 읽는 방향으로 점진적으로 단순화할 수 있습니다.

### 4. 기존 launch 명령 유지

- `agribot_bringup/launch/simulation.launch.py`에 exporter 노드를 추가했습니다.
- 따라서 사용자는 기존 실행 명령을 바꾸지 않아도 됩니다.

```bash

source install/setup.bash

```

### 5. 패키지 의존성과 테스트 보강

- `package.xml`에 `ament_index_python`, `nav_msgs`, `tf2_ros` 의존성을 추가했습니다.
- `setup.py`에 `runtime_snapshot_exporter` 엔트리포인트를 등록했습니다.
- `test_runtime_snapshot_service.py`를 추가했고, `test_simulation_launch.py`도 새 노드가 포함된 기준으로 갱신했습니다.

## [S14P-104] [ROS] 웹 수동 명령 Nav2·순찰 제어 runtime executor 구축

### 개요

S14P-104 웹 수동 명령 runtime executor를 추가했습니다.

이번 MR의 목적은 다음 3가지입니다.

1. backend가 `/tmp/agribot_runtime/robot_manual_command.json`에 남긴 웹 명령을 ROS 런타임에서 실제 동작으로 실행한다.
2. `navigate_to_pose`, `pause_patrol`, `resume_patrol`, `return_home`를 기존 Nav2 및 patrol 서비스 흐름에 자연스럽게 연결한다.
3. 실행 결과를 `/tmp/agribot_runtime/robot_manual_command_status.json`에 기록해 backend가 같은 runtime snapshot 패턴으로 읽을 수 있게 한다.

## [S14P-105] [BE] 로봇 명령 파일 브리지와 실데이터 zones API 연결

### 개요

backend의 `/api/v1/robot/commands`를 더 이상 단순 `accepted: true` 응답으로 끝내지 않고, 기존 runtime snapshot 아키텍처와 같은 파일 브리지 방식으로 실제 ROS 런타임에 전달되도록 정리했습니다.

이번 작업의 목적은 다음과 같습니다.

1. backend가 `/tmp/agribot_runtime/robot_manual_command.json`을 원자적으로 기록해 agribot executor가 바로 읽을 수 있게 한다.
2. executor가 남긴 `/tmp/agribot_runtime/robot_manual_command_status.json`을 API로 다시 노출해 프론트와 운영자가 최신 명령 상태를 확인할 수 있게 한다.
3. zones API를 현재 맵 구조와 patrol waypoint 기준의 실데이터로 바꿔 `move_to_zone`이 실제 대표 pose를 기준으로 동작하게 한다.

이번 작업으로 backend는 ROS를 직접 import하지 않으면서도 `navigate_to_pose`, `pause_patrol`, `resume_patrol`, `return_home`, `move_to_zone`을 모두 같은 명령 체계로 runtime bridge에 연결할 수 있게 됐습니다.

---

### 배경

S14P21A602-104까지 진행한 시점에는 agribot_ws 쪽 executor가 이미 `/tmp/agribot_runtime/robot_manual_command.json`을 읽고 실제 Nav2 및 patrol 서비스로 연결할 수 있는 상태였습니다.

하지만 backend에는 아래 빈 부분이 남아 있었습니다.

- `/api/v1/robot/commands`가 실제 파일을 쓰지 않고 단순 accepted 응답만 반환
- executor가 기록한 최신 status 파일을 다시 읽는 API 부재
- `/api/v1/zones`가 현재 맵 구조와 무관한 stub 데이터만 반환
- `move_to_zone`이 실제 zone representative pose가 아니라 별도 체계처럼 남아 있음

즉, backend도 pose snapshot과 semantic layer snapshot을 읽는 기존 구조와 같은 결로, **REST 요청 -> command file write -> ROS executor -> status file read** 흐름을 완성할 필요가 있었습니다.

## [S14P-106] [FE] 로봇 지도 클릭 이동과 실데이터 구역 이동 UI 연결

### 개요

`/robot` 페이지에서 실제 지도 위 빈 공간을 클릭해 이동 목표를 지정하고, `/zones`의 실데이터를 이용해 빠른 구역 이동도 바로 요청할 수 있도록 프론트엔드 관제 화면을 완성했습니다.

이번 작업의 목적은 다음과 같습니다.

1. `/robot/map`, `/robot/map/raw`, `/robot/map/layers`를 이용해 실제 occupancy map 기반 지도 렌더링으로 정리한다.

## [S14P-108] [FE] /robot 페이지에 새 목적지 선점 전환 피드백 추가

### 개요

`/robot` 페이지에서 지도 클릭으로 이동 명령을 보낸 뒤, 이동 중 다시 새 목적지를 지정하면 사용자가 "이전 목표 대신 새 목표로 전환됐다"는 점을 바로 이해할 수 있도록 프론트 상태 카드와 안내 문구를 보강했습니다.

이번 작업의 목적은 아래 3가지입니다.

1. 새 좌표나 구역 이동 요청이 기존 이동을 선점하는 경우, 즉시 UI에서 그 사실을 보여준다.
2. polling이 아직 최신 command status를 받기 전에도 사용자가 전환 요청이 반영됐다는 피드백을 볼 수 있게 한다.
3. `/robot/commands/latest`가 내려주는 `preempt_current_navigation` 계약을 프론트 타입과 상태 카드에 연결한다.

---

### 배경

backend와 executor는 이미 새 이동 명령이 현재 순찰 또는 수동 이동을 선점하도록 바뀌어 있었지만, 프론트는 여전히 `running` 상태를 모두 같은 "이동 실행 중"으로만 보여주고 있었습니다.

그래서 실제로는 새 목표로 갈아탄 상황이어도 사용자는 아래처럼 느끼기 쉬웠습니다.

- 방금 새 목적지를 눌렀는데, 화면에는 예전 이동과 다른 점이 잘 안 보임
- 상태 카드가 polling을 한 번 더 돌기 전까지는 "정말 새 목표가 반영됐는지" 확신하기 어려움
- `좌표 직접 지정`만 보이고 새 목표 좌표나 선점 전환 사실이 충분히 드러나지 않음

즉, 기능은 이미 동작해도 사용자 피드백이 약해서 시연성과 조작 신뢰감이 떨어지는 상태였습니다.

---

### 주요 변경 사항

1. API 타입과 파싱 보강
- `frontend/src/lib/api/agribot.ts`에서 `RobotCommandStatus`, `RobotCommandDispatch` 타입에 `preemptCurrentNavigation`을 추가했습니다.
- `/robot/commands/latest`와 `/robot/commands` 응답에서 `preempt_current_navigation` 값을 읽어 프론트 상태 판단에 사용할 수 있게 했습니다.

2. `/robot` 페이지 상태 관리 보강
- `MapControlPage`에 `transitionFeedback` 상태를 추가했습니다.
- 새 이동 요청을 보낼 때 이미 다른 이동이 `pending` 또는 `running`이면, 이번 요청이 선점 전환이라는 점을 로컬 상태로 즉시 기록하도록 했습니다.
- 이 덕분에 polling이 아직 새 `command_id`를 가져오기 전에도 "기존 이동을 중단하고 새 목표로 전환 중"이라는 메시지를 바로 띄울 수 있습니다.

3. 상태 카드 문구 개선
- 기존 `commandStatusCopy`는 `pending`, `running`을 모두 일반 이동으로만 표현했지만, 이제 선점 전환 상태일 때는 `새 목표 전환 요청`, `새 목표 전환 준비`, `새 목표로 전환 중`처럼 더 구체적으로 보여줍니다.
- 상태 카드 아래에 `새 목적지 우선 적용` 안내 박스를 추가해, 어떤 좌표 또는 구역으로 갈아탔는지 한 번 더 명시했습니다.
- detail 카드에도 `최근 목표 좌표`를 추가해 지도 클릭 이동에서 실제 새 좌표를 바로 확인할 수 있게 했습니다.

4. notice 문구 조정
- 새 이동 요청이 기존 이동을 선점하는 경우, 하단 notice도 일반 "이동 요청을 보냈습니다"가 아니라 "기존 이동을 중단하고 새 목표로 전환 중입니다" 기준으로 보여주도록 바꿨습니다.
- 선점 전환이 성공적으로 끝나면 완료 notice도 새 목표 기준으로 정리해서 보여줍니다.

## [S14P-137] [ROS/Gazebo] 수확 시뮬레이션·Gazebo 자산·안전 종료 로직만 선별 반영

### 개요

`feature/S14P21A602-137` 브랜치 전체를 `dev2`에 합치지 않고, 실제로 필요한 범위만 골라서 반영했습니다.

이번 작업의 핵심은 다음 4가지입니다.

1. Gazebo 수확 시뮬레이션 강화 로직만 선별 반영
2. 로봇 모델, 월드, state bridge 등 Gazebo 실행 품질에 직접 영향을 주는 자산만 선별 반영
3. `Ctrl+C` 종료 시 과도한 프로세스 정리로 Ubuntu 세션까지 건드릴 수 있는 종료 정리 로직 개선 반영
4. 선별 merge만으로는 들어오지 않는 `startup_map_tf_broadcaster.py` 수정사항을 `dev2`에 직접 패치

즉, 프론트 대규모 변경, backend AI judgment 계층, 문서 정리, git hook 등은 제외하고, Gazebo/ROS2 실행 안정성과 수확 시연 품질에 필요한 부분만 옮겼습니다.

---

### 배경

`feature/S14P21A602-137`에는 유용한 변경이 많았지만, 그대로 전체 merge를 하면 아래와 같은 문제가 있었습니다.

- 프론트 대규모 화면/스타일 변경까지 한 번에 들어와 범위가 지나치게 커짐
- backend AI judgment, 문서 삭제/정리, 기타 스크립트까지 함께 들어와 회귀 위험이 커짐
- 원하는 목적은 Gazebo 수확 시뮬레이션, 로봇 모델/월드, 종료 안정화, TF/sim-time 문제 완화였는데 전체 merge는 이 목적보다 훨씬 넓음

또한 TF/sim-time 관련 진단 결과, 실제 증상의 핵심 중 하나였던 `startup_map_tf_broadcaster.py`는 `dev2..feature/S14P21A602-137` 범위에 포함되지 않아, 단순 merge로는 해결되지 않았습니다.

그래서 이번 작업은 **선별 반영 + dev2 직접 패치** 방식으로 진행했습니다.

### 주요 변경 사항

1. Gazebo 수확 시뮬레이션 관련 파일만 선별 반영했습니다.

- `harvest_action_server.py`의 수확 애니메이션 강화 로직
- `harvest_route_contract.py` 신규 추가
- `harvest_route_node.py` 개선
- `harvest_simulation.py` 신규 추가
- harvest 관련 테스트와 recovery BT XML 수정

이 변경으로 수확 시퀀스, 경로 계약, 시뮬레이션 검증 코드가 `dev2`에도 들어오도록 정리했습니다.

2. 로봇 모델/월드/Gazebo 자산 관련 변경만 선별 반영했습니다.

- `agribot` 모델 SDF/URDF 수정
- `farm_world.sdf` 수정
- 토마토 Gazebo mesh 및 model 경량 자산 반영
- `ros_gz_state_bridge.yaml`, `ros_gz_state_bridge_lite.yaml` 신규 추가
- spawn/world 관련 테스트 추가

이 변경으로 Gazebo 쪽 로봇팔 장착/방향, 작물 자산, 상태 브리지 설정을 `dev2` 기준으로 맞췄습니다.

3. `Ctrl+C` 종료 시 과도한 프로세스 정리 문제를 줄이기 위한 shutdown cleanup 로직을 선별 반영했습니다.

- `launch_profile.py` 신규 추가
- `shutdown_cleanup.py` 신규 추가
- `shutdown_cleanup_cli.py` 신규 추가
- `simulation.launch.py`와 관련 테스트 수정
- `cleanup_sim_processes.sh`, `agribot_launch.sh`, `agribot_env.sh` 조정

이 로직은 “우리 시뮬레이션 세션이 띄운 대상만 정리한다”는 방향으로 정리되어, 로그아웃/데스크톱 세션에 영향을 줄 수 있는 과도한 종료를 줄이도록 의도되어 있습니다.

4. `startup_map_tf_broadcaster.py`는 `dev2`에 직접 수정했습니다.

선별 merge만으로는 이 파일이 바뀌지 않으므로, 아래 동작을 직접 넣었습니다.

- `/amcl_pose`를 한 번이라도 받으면 임시 identity `map -> odom` broadcaster를 중지
- 타이머를 cancel/destroy 하고, 관련 subscription도 정리
- AMCL이 takeover한 이후에도 임시 TF가 계속 남아 frame 경쟁이나 `TF_OLD_DATA`를 만들지 않도록 수정

이 부분은 단순 copy & paste로도 해결 가능하지만, **`dev2` 현재 코드 문맥에 맞게 타이머와 subscription 정리까지 함께 넣어야** 완결됩니다. 이번 작업에서는 그 정리까지 포함해 직접 반영했습니다.

5. 이번에 새로 생성된 파일에는 한글 주석/설명을 추가했습니다.

비전공자 팀원도 신규 파일의 역할을 빠르게 이해할 수 있도록, 신규 Python/YAML/테스트 파일에 “왜 이 파일이 필요한지” 중심의 한글 설명을 넣었습니다.

6. 후속으로 시연 실행 문서도 현재 `dev2` 기준에 맞게 짧게 바로잡았습니다.

- `npm ci`를 언제 다시 해야 하는지
- `npm run build`가 기본 필수인지 아닌지
- `frontend/dist`, ROS `build/install/log`를 언제 지워야 하는지

를 문서에 짧게 정리해, 시연 직전 불필요한 재설치/재빌드로 시간을 쓰지 않도록 맞췄습니다.

7. 후속 보정으로 `기존 dev2`의 주행 시작 경로도 다시 기본값에 맞춰 복원했습니다.

- `launch_profile.py`의 `balanced` 시뮬레이션 기본값에서 `use_runtime_support`를 다시 `true`로 조정
- 이로 인해 `simulation.launch.py` 기본 실행 시 `runtime_snapshot_exporter`, `robot_manual_command_executor`, `mission_bridge_executor`가 다시 함께 올라오도록 복원
- 프론트에서 수확/이동 버튼을 눌렀을 때 backend가 기록한 runtime request 파일이 ROS 토픽 publish 단계로 이어지지 않던 회귀 가능성을 줄임

추가로 `operator_runtime_health.sh`에도 `AGRIBOT_GRAPHICS_PROFILE` 점검을 넣어, GPU/graphics auto 선택이 실제 프로세스에 일관되게 반영됐는지 확인할 수 있게 했습니다.

8. 후속 보정으로 `simulation.launch.py`의 Gazebo/Nav2 기동 안정화 요소도 다시 가져왔습니다.

- session별 `gz_partition` 값을 navigation/localization 쪽에도 같이 전달
- Gazebo world가 paused 상태로 남는 경우를 막기 위한 unpause 타이머 2회 복원
- Nav2 지연 기동 시간을 다시 늘려, GUI/Gazebo/bridge가 안정화되기 전에 첫 주행 요청이 묻히는 회귀 가능성을 줄임

이 단계는 “버튼을 눌렀는데 명령은 들어갔지만 실제 주행이 안 붙는” 증상에 대응하기 위한 보정입니다.

9. 직접 재현 검증 끝에, simulation 모드의 TF 기준선도 `기존 dev2` 방식으로 복원했습니다.

실제 backend API로 수확 요청을 넣어 확인한 결과,

- `mission_bridge_executor`는 정상적으로 요청을 받아 `harvest_route_node`까지 전달함
- `harvest_route_node`도 실제로 접근 목표를 Nav2에 보냄
- 하지만 `startup_map_tf_broadcaster`가 빠진 뒤 `map <- odom` lookup이 `future/past extrapolation`으로 깨지며 Nav2가 abort

하는 것이 원인이었습니다.

그래서 아래를 반영했습니다.

- `simulation.launch.py`에서 `publish_map_to_odom_tf`를 다시 `true`로 복원
- `navigation.launch.py`, `localization.launch.py`에 `use_startup_map_tf_broadcaster` 인자를 다시 도입
- simulation 모드에서는 `startup_map_tf_broadcaster`를 끄고, 고정 `map -> odom` TF를 유지

이 조합으로 같은 수확 요청을 다시 넣었을 때,

- `Harvest route planned`
- `Begin navigating ...`
- `Passing new path to controller`
- backend mission status `running`

까지 직접 확인했습니다.

- `py_compile` 대상 파일 모두 통과

- 후속 보정으로 수정한 `navigation.launch.py`, `localization.launch.py` 문법 검증 통과
- 직접 재현 시 이전에는 `map <- odom` TF 외삽 오류로 abort 되던 수확 요청이, 보정 후에는 `mission status=running`과 controller path 추종 로그까지 확인됨
- 나머지 ROS launch/navigation 계열 테스트는 현재 셸 환경에 `launch.actions`, `action_msgs`, `ament_index_python` 등이 없어 import 단계에서 중단됨

즉, **선별 반영한 코드의 기본 문법과 Gazebo 자산 검증은 통과**했고, ROS 전용 테스트는 워크스페이스 sourcing이 된 ROS 실행 환경에서 추가 확인이 필요합니다.

### 확인된 이슈 및 주의사항

1. 이번 작업은 의도적으로 전체 merge를 피했습니다.

- 프론트 변경
- backend AI judgment 저장/후처리
- 문서/도구 정리

위 범위는 이번 MR에 포함하지 않았습니다. 따라서 `feature/S14P21A602-137`의 모든 변경이 들어온 것은 아닙니다.

2. `startup_map_tf_broadcaster.py`는 branch merge로 들어온 것이 아니라 `dev2` 직접 패치입니다.

따라서 “137 브랜치 선별 merge 결과”만 비교하면 보이지 않을 수 있으니, 실제 반영 여부는 파일 내용을 기준으로 확인해야 합니다.

3. ROS 테스트는 실행 환경 영향을 크게 받습니다.

`source /opt/ros/<distro>/setup.bash` 와 워크스페이스 overlay가 없는 셸에서는 일부 테스트가 import 단계에서 실패할 수 있습니다.

## [leejaeyup_final] 최종 시연용 관측 접근 보완과 Gazebo 자산 정렬 마무리

### 개요

`dev2`에서 누적하던 시연 안정화 작업을 최종 정리 브랜치인 `leejaeyup_final`로 분기하고, 마지막으로 남아 있던 미커밋 변경까지 묶어 원격에 올릴 수 있는 상태로 정리했습니다.

이번 정리의 핵심은 다음 5가지입니다.

1. 울타리 근처 최종 관측 지점이 막혀도 더 가까운 대체 지점을 순차 탐색하도록 수동 명령 실행기를 보강
2. 지도 표시용 접근 좌표를 통로 방향(`lane_side`)별로 다르게 계산하도록 backend/경로 설정을 확장
3. 로봇팔이 상하 반전된 상태로 보이던 Gazebo/URDF 자산을 동일 기준으로 재정렬
4. Ogre2 환경에서 토마토가 흰색으로 보이지 않도록 모델 재질 검증값을 정리
5. 시연 명령어 문서와 트러블슈팅 문서를 보강해 비전공자 팀원도 시연 절차와 장애 대응을 따라갈 수 있게 정리

---

### 배경

기존 2단계 이동 구조만으로도 대부분의 작물 접근은 가능했지만, 울타리와 너무 가까운 목표는 마지막 관측 지점으로 가는 경로가 막혀 실패 처리되는 경우가 남아 있었습니다.

또한 실제 시연 품질 측면에서는 다음 문제가 계속 남아 있었습니다.

- 로봇팔 메쉬와 충돌체 방향이 어긋나 수확 장면이 어색함
- 특정 렌더러에서 토마토가 흰색으로 보여 발표 품질이 떨어짐
- 중앙 통로 작물은 프론트 지도 마커가 실제 의도보다 멀게 찍혀 조작 맥락이 불명확함
- 시연 준비 절차와 장애 대응 경험이 문서로 충분히 축적되지 않아 팀원별 편차가 생길 수 있음

그래서 이번 작업에서는 **주행 마무리 안정화 + Gazebo 자산 정렬 + 시연 문서화**를 한 번에 정리했습니다.

---

### 주요 변경 사항

1. **최종 관측 대체 지점 순차 탐색 추가**
   - `robot_manual_command_executor.py`에 최종 관측 지점까지의 중간 후보를 계산하는 보조 함수를 추가했습니다.
   - 경로 탐색 실패 시 바로 최종 실패로 가지 않고, 울타리 쪽으로 가능한 최대 근접 지점을 여러 개 시도한 뒤 성공 가능한 후보를 선택하도록 바꿨습니다.
   - 실제로 최종 목표 대신 조정된 대체 지점으로 도착했을 때는 상태 메시지도 그에 맞게 바뀌도록 정리했습니다.
   - 관련 단위 테스트를 추가해 후보 생성 순서와 작은 거리에서는 후보를 만들지 않는 조건을 검증했습니다.

## [main-주석] [Frontend/Navigation] 작물 접근 후보 선택을 최근접 기준에서 통로 비용 기준으로 바꾸고 후보 전체를 executor에 전달

### 개요

이번 작업은 토마토 좌우 어느 쪽으로 접근할지 고를 때, 화면에서 단순히 "가까워 보이는 좌표"를 고르지 않고 실제 통로를 따라 덜 돌아가는 후보를 우선 고르도록 바꾼 작업입니다.

기존에는 프론트엔드가 관측 후보 여러 개 중 하나를 너무 빨리 확정해서 보내는 경우가 있었고, 이 때문에 현재 로봇 위치에서는 오른쪽 후보가 더 자연스러운데도 왼쪽 후보를 먼저 고정해 보내는 상황이 생길 수 있었습니다.

이번 변경의 핵심은 2가지입니다.

1. 프론트엔드 기본 선택 전략을 `nearest`에서 `route-cost`로 변경
2. 농장 명령 화면에서는 후보 하나만 보내지 않고, 좌/우 후보 전체를 executor에 함께 보내 백엔드가 waypoint 그래프 비용 기준으로 최종 선택

즉, "지금 눈으로 보기에 가까운 점"보다 "실제로 통로를 따라 이동할 때 덜 돌아가는 점"을 더 우선하게 만들었습니다.

---

### 배경

토마토는 왼쪽 통로에서도 볼 수 있고 오른쪽 통로에서도 볼 수 있는 경우가 있습니다.

이때 단순 직선거리만 보면 두 후보가 비슷하거나, 오히려 실제 주행과 반대로 보일 수 있습니다.

예를 들어:

- 현재 로봇이 이미 오른쪽 통로 쪽에 가까이 와 있는데
- 토마토 중심 좌표만 보면 왼쪽 후보가 조금 더 가까워 보일 수 있습니다.

하지만 실제 로봇은 통로와 웨이포인트를 따라 움직이기 때문에, "직선거리"보다 "통로를 따라 꺾어 가며 누적한 이동량"이 더 중요합니다.

이번 작업에서 말하는 **그래프 비용**은 이 누적 이동량을 뜻합니다.

- 직선으로 2m + 위로 3m 가야 하면 비용은 5m
- 눈으로는 가까워 보여도 통로를 두 번 갈아타야 하면 비용이 더 커짐

즉, 그래프 비용은 "실제로 길을 따라 갔을 때 얼마나 많이 움직여야 하는가"를 수치로 본 것입니다.

---

### 주요 변경 사항

1. 작물 접근 후보 선택 전략에 `route-cost`를 추가했습니다.

- 파일: `frontend/src/lib/robot-map/approach-pose.ts`
- 기존 `nearest`, `harvest-primary` 외에 `route-cost` 전략을 추가했습니다.
- `route-cost`는 후보 navigation pose까지의 이동을 단순 대각선 직선거리로 보지 않고, x축/ y축 선분을 따라 누적한 이동량으로 계산합니다.

이 계산은 현재 프론트엔드가 전체 waypoint 그래프를 직접 들고 있지 않은 상태에서, 실제 통로 주행 느낌을 가장 가깝게 반영하려는 근사치입니다.

2. 기본 작물 계획 생성 전략을 `route-cost`로 변경했습니다.

- 파일: `frontend/src/lib/robot-map/plant-navigation-plan.ts`
- 별도 옵션을 주지 않으면 이제 기본적으로 `route-cost` 기준으로 관측 후보를 고릅니다.

즉 앞으로는 지도 클릭, 개발자 진단 화면, 작물 명령 화면에서 공통적으로 같은 기본 판단 기준을 쓰게 됩니다.

3. 농장 명령 화면은 후보 하나를 잘라 보내지 않도록 바꿨습니다.

- 파일: `frontend/src/pages/farm-command-page.tsx`
- 기존에는 `selectedCandidateOnly: true`라서 프론트엔드가 고른 후보 1개만 executor에 전달했습니다.
- 이제는 `selectedCandidateOnly: false`로 바꿔, 좌/우 후보 전체를 함께 전달합니다.

이렇게 바꾸면 실제 실행 직전의 현재 위치를 가장 잘 아는 백엔드 executor가,

- 현재 pose
- patrol waypoint 그래프
- 후보 inspect waypoint 목록

을 기준으로 "실제로 덜 돌아가는 후보"를 다시 고를 수 있습니다.

4. 개발자 진단 페이지도 같은 기준을 보도록 맞췄습니다.

- 파일: `frontend/src/pages/develop-page.tsx`
- 개발자 페이지에서도 `nearest` 대신 `route-cost`를 쓰도록 바꿨습니다.

이제 개발자 페이지에서 보는 계획과 실제 executor가 최종 선택하는 방향이 전보다 덜 어긋납니다.

---

### 기대 효과

1. 토마토 좌/우 접근 후보가 있는 경우, 단순 최근접보다 실제 주행이 자연스러운 쪽을 더 자주 선택합니다.
2. 농장 명령 화면이 후보를 너무 일찍 확정하지 않으므로, 현재 로봇 위치가 바뀐 상태에서도 executor가 더 나은 후보를 고를 수 있습니다.
3. "왜 저쪽으로 먼저 가는지"가 줄고, 좌/우 후보 선택 기준이 통로 구조에 더 맞게 정리됩니다.
4. 앞으로 waypoint 그래프가 바뀌더라도, 프론트에서 후보를 억지로 1개만 고정하지 않아 미래 확장에도 덜 방해됩니다.

## [S14P-주석] [ROS/Nav] crop-side pose에서 다음 작물로 갈 때 큰 우회를 줄이도록 통로 그래프와 시작점 선택 로직 개선

### 개요

23번 토마토에 도착한 뒤 16번이나 13번 같은 다른 토마토를 다시 선택하면, 로봇이 먼저 북쪽 entry/turn 쪽으로 크게 우회한 뒤 내려오거나, 통로를 갈아타는 기준점이 어색하게 잡히는 문제가 있었습니다.

이번 작업은 단순히 속도나 yaw를 더 깎는 미세 튜닝이 아니라, **작물 옆 최종 관측 자세(crop-side pose)에서 다음 목표를 누를 때 어떤 통로로 어떻게 복귀할지**를 다시 설계한 작업입니다.

핵심은 아래 2가지입니다.

1. 같은 작물 행(y가 같은 inspect waypoint) 사이의 횡이동을 waypoint 그래프에 정식으로 표현
2. 시작 waypoint를 순서형 휴리스틱으로 하나씩 찍는 대신, 후보 시작점들의 **전체 경로 비용**을 비교해서 선택

즉, 기존처럼 “목표 lane이 다르니 일단 그 lane의 북/남 entry로 가자”가 아니라,  
“지금 위치에서 같은 행 통로로 먼저 복귀한 뒤, 필요한 row까지 직진하고, 같은 행에서 옆 lane으로 갈아타는 쪽이 실제로 더 짧고 더 자연스러운가?”를 코드가 직접 비교하게 바꿨습니다.

---

### 배경

지금까지는 세그먼트 실행기 자체는 `spin -> drive_on_heading` 기반이라 직진/회전 중심으로 잘 동작할 수 있었지만, 그 전에 계산되는 waypoint 경로가 이미 이상한 경우가 있었습니다.

대표적으로 23번 토마토 옆 crop-side pose 근처에서는:

- 현재 위치가 waypoint 그래프 위가 아님
- `route_egress` 후보는 잘 계산돼도, target lane이 다르다는 이유로 버려질 수 있었음
- 그 결과 target lane의 북쪽/남쪽 entry/turn이 시작점으로 선택돼 큰 우회가 생겼음

그래서 로봇이 “직선 주행을 못 하는 것”이 아니라,  
**잘못 고른 통로 진입점을 향해 직선 주행을 하고 있었던 것**에 가까웠습니다.

이번에는 이 구조 자체를 손봐서, 23번에서 16번/13번처럼 연속 이동할 때도 경로가 더 사람 눈에 자연스럽게 보이도록 바꿨습니다.

---

### 주요 변경 사항

1. `manual_navigation_routing.py`에 같은 행 inspect waypoint 간 횡이동 edge를 추가했습니다.

- `build_waypoint_adjacency(...)`에서 기존 route 내부 연결과 북/남 connector 연결만 쓰던 구조에,
  같은 y 높이 inspect waypoint끼리의 lateral transfer edge를 추가했습니다.

## [S14P-139] [ROS/Nav2] 최종 토마토 접근도 세그먼트 상태기로 통일해 마지막 회전 뒤 직진 패턴을 유지하도록 개선

### 개요

이전 리팩토링으로 통로 진입과 중간 경유 구간은 `Spin + DriveOnHeading` 기반 세그먼트 상태기로 바뀌었지만, **토마토 바로 앞 최종 접근 구간은 여전히 `NavigateToPose`로 실행되고 있었습니다.**

그래서 실제 주행을 보면

- 첫 직진은 예전보다 반듯해졌는데
- 목표 토마토 근처에 와서 방향을 바꾸는 순간
- 다시 좌우로 여러 번 흔들리거나
- 곡선 보정처럼 보이는 움직임이 나타나는

문제가 남아 있었습니다.

이번 작업에서는 그 마지막 구간까지 같은 segmented route 구조로 통일해, **"직진 -> 제자리 회전 -> 다시 직진"** 패턴이 마지막 접근에서도 유지되도록 정리했습니다.

---

### 배경

그동안 주행 품질이 완전히 만족스럽지 않았던 이유는, 구조가 아래처럼 반쯤만 바뀌어 있었기 때문입니다.

1. 통로 이동 / route anchor 이동
   - `spin` + `drive_on_heading`
   - 사람이 보기에도 상용 로봇처럼 비교적 깔끔한 패턴

2. final observation 이동
   - 다시 `NavigateToPose`
   - 로컬 컨트롤러가 부드러운 곡선 보정 경로를 만들며 움직임

즉 "어디까지는 새 구조, 마지막은 예전 구조"가 섞여 있어서, 사용자가 가장 눈에 띄게 보는 토마토 앞 마지막 회전 구간에서 다시 예전 문제처럼 보였던 것입니다.

이번 수정의 목적은 아주 명확합니다.

- 세그먼트 상태기를 첫 구간에만 쓰지 않고
- 최종 토마토 접근에도 똑같이 적용해서
- 마지막 회전 뒤에도 곧바로 직선 접근 패턴을 유지하게 만든다

동시에, 세그먼트 실행이 여건상 실패할 때는 **기존 Nav2 fallback**이 남아 있도록 유지해 안정성은 깨지지 않게 했습니다.

---

### 주요 변경 사항

1. `_dispatch_final_observation_goal(...)`의 직접 `NavigateToPose` 실행을 제거했습니다.

- 파일: `S14P21A602/agribot_ws/src/agribot_bringup/agribot_bringup/robot_manual_command_executor.py`

이전에는 final observation 단계에 들어오면,

- probe 결과로 안전하다고 판단된 최종 관측 pose를 받더라도
- 결국 `_dispatch_final_observation_goal(...)` 안에서
- `NavigateToPose.Goal()`을 직접 만들고
- 그 순간부터 다시 일반 Nav2 경로 추종으로 내려갔습니다.

이번에는 이 구조를 없애고, final observation도 기존 `_dispatch_navigation_goal(...)`로 넘기도록 바꿨습니다.

2. 최종 관측 단계도 segmented route 진입 함수로 통일했습니다.

이제 final observation 단계에서:

- `context.navigation_phase = FINAL_OBSERVATION`
- `context.target_pose = 최종 관측 pose`

까지는 동일하게 맞춘 뒤,

- `_dispatch_navigation_goal(..., prefer_behavior_plan=True)`

로 넘겨서,

- 가능하면 `spin -> drive_on_heading` 세그먼트 계획을 만들고
- 그 계획이 유효하면 그대로 segmented execution
- 아니면 기존 Nav2 목표 추종으로 fallback

흐름을 타게 만들었습니다.

3. 결과적으로 마지막 접근 구간도 "회전과 직진을 분리한 행동"으로 맞췄습니다.

이 변경 뒤 기대하는 동작은 아래와 같습니다.

- 통로를 따라 직진
- 목표 토마토 접근 방향으로 멈춘 상태에서 회전
- 회전이 끝나면 다시 짧은 직선 접근

즉 예전처럼 "최종 접근만 다시 곡선 보정으로 흘러가는" 혼합 구조를 줄이는 것이 핵심입니다.

4. fallback 구조는 그대로 유지했습니다.

이번 수정은 공격적인 구조 정리이지만, 여전히 안전장치는 남겨 두었습니다.

- segmented route 생성이 안 되면
- spin / drive_on_heading action server가 없으면
- 실행 중 실패하면

기존 `_dispatch_navigation_goal(...)` 내부 fallback을 통해 일반 Nav2 목표 추종으로 복귀합니다.

즉, 이번 수정은 "예쁘게 보이도록 덧칠"이 아니라,

- 우선순위 1: 회전/직진 분리 실행
- 우선순위 2: 실패 시 기존 안전 경로 유지

구조로 짜 둔 것입니다.

- 목표 근처에 도착한 뒤 제자리에서 고개를 더 돌리며
- 사용자가 보기에는 "왜 이미 왔는데 아직도 탐색하지?"처럼 느껴지는 문제가 남아 있었습니다.

이번 작업에서는 이 원인을 **최종 도착을 "좌표 + 최종 yaw"로 보고 있었던 구조**로 판단했고, 이를 **"좌표 도착 우선" 구조**로 바꿨습니다.

즉, 이제는 가능한 경우

- 한 번 회전해서 접근 방향을 정하고
- 직선으로 도착한 뒤
- 그 자세 그대로 성공 처리

가 되도록 정리했습니다.

---

### 배경

이번 문제는 단순히 마지막 접근이 segmented route가 아니어서만 생긴 것이 아니었습니다.

실제 코드 기준으로 보면 아래 2가지가 남아 있었습니다.

1. segmented route planner가 마지막에 `target_pose.yaw`까지 맞추기 위한 추가 spin을 붙일 수 있었다.
2. `build_intermediate_final_observation_targets(...)`가 route anchor와 final target 사이 중간 probe target을 만들 때, 좌표뿐 아니라 yaw도 함께 보간하고 있었다.

이 구조에서는 로봇이 좌표상 거의 도착했더라도

- "아직 목표 yaw를 덜 맞췄다"
- "다음 probe 지점 yaw가 조금 더 바뀌어 있다"

고 해석할 수 있어서, 사용자는 이를 `두리번거림`처럼 보게 됩니다.

하지만 현재 프로젝트의 목표는

- 토마토 앞에 정확히 서는 것보다
- 통로형 환경에서 직선 위주로 깔끔하게 접근하고
- 도착 후 추가 제자리 회전 없이
- "목적지에 도달했습니다."를 안정적으로 띄우는 것

에 더 가깝습니다.

그래서 이번 수정에서는 **최종 yaw 정렬보다 목표 좌표 도착을 우선**하는 방향으로 정책을 바꿨습니다.

---

### 주요 변경 사항

1. segmented route planner에 `enforce_final_yaw` 옵션을 추가했습니다.

- 파일: `S14P21A602/agribot_ws/src/agribot_bringup/agribot_bringup/manual_navigation_routing.py`

기존 `build_manual_navigation_behavior_plan(...)`은 마지막 drive 뒤에 `target_pose.yaw`까지 맞추기 위한 spin을 항상 추가할 수 있었습니다.

이번에는 `enforce_final_yaw` 옵션을 넣어,

- 필요하면 기존처럼 마지막 yaw까지 맞추고
- 끄면 마지막 직진이 끝난 자세 그대로 도착 성공으로 처리

할 수 있게 바꿨습니다.

2. executor 기본 정책을 "도착 후 final yaw 강제 안 함"으로 바꿨습니다.

- 파일: `S14P21A602/agribot_ws/src/agribot_bringup/agribot_bringup/robot_manual_command_executor.py`

추가 파라미터:

- `route_behavior_enforce_final_yaw` (기본값 `False`)

즉 현재 기본 실행 정책은

- 목표 좌표까지는 segmented route로 접근
- 마지막에 굳이 제자리에서 목표 yaw까지 다시 맞추지 않음

입니다.

이 설정은 나중에 팀이 필요하면 다시 켤 수 있도록 파라미터로 남겨 두었습니다.

3. 최종 관측 중간 probe target의 yaw를 선형 보간하지 않도록 바꿨습니다.

기존 `build_intermediate_final_observation_targets(...)`는

- route anchor yaw
- final target yaw

사이를 fraction 비율로 섞어서 intermediate target yaw를 만들었습니다.

이 방식은 좌표는 직선이어도, 로봇이 각 intermediate target마다 조금씩 다른 yaw를 목표로 보게 만들어 마지막 접근을 불안정하게 만들 수 있었습니다.

이번에는 중간 target의 yaw를 모두 **실제 접근 직선 방향 heading**으로 통일했습니다.

즉 route anchor에서 final target으로 가는 선분 방향이 east면,

- intermediate probe target들도 모두 east heading을 공유하도록 바꿨습니다.

4. 성공 메시지를 "목적지에 도달했습니다." 중심으로 바꿨습니다.

기존에는 상황에 따라

- `이동 명령이 완료되었습니다.`
- `최종 관측 위치까지 이동을 완료했습니다.`

처럼 다소 기술적인 문구가 기록됐습니다.

이번에는 프론트엔드가 그대로 읽어 보여 줄 때도 더 자연스럽도록, 성공 메시지를 `목적지에 도달했습니다.` 중심으로 정리했습니다.

울타리 제약 때문에 가장 가까운 근접 위치까지만 들어간 경우에는 그 이유를 뒤에 덧붙이도록 유지했습니다.

5. 회귀 테스트를 추가했습니다.

- final yaw가 달라도 `enforce_final_yaw=False`일 때 마지막 spin이 생기지 않는지
- intermediate final observation target이 접근 선분 heading을 유지하는지

를 테스트로 고정했습니다.

이렇게 해 두면 이후 팀원이 다른 주행 개선을 하더라도,

- 다시 도착 후 제자리 회전이 붙거나
- 중간 yaw 보간이 부활하는

회귀를 빨리 잡을 수 있습니다.

## [S14P-138] [ROS/Nav2] 가까운 토마토 목표는 route anchor를 생략하고 최종 관측 위치 직행을 먼저 시도하도록 개선

### 개요

프론트엔드 개발자 페이지(`/develop`)에서 23번 토마토 같은 목표를 눌러 보면, 실제 실행기는 `finalTargetPose=(1.85, 6.0)`이 아니라 먼저 `routeTargetPose=(0.0, 6.0)`인 `route_anchor` 단계로 들어가고 있었습니다.

이 구조 때문에 로봇은 가까운 토마토도 바로 향하지 않고,

1. 중간 통로 anchor를 먼저 향해 꺾고
2. anchor에 도착한 뒤
3. 다시 최종 관측 위치로 들어가는

두 단계 주행을 기본값처럼 수행했습니다.

이번 작업에서는 이 로직을 전면 제거하지 않고, **가까운 2단계 목표에 대해서만 먼저 "최종 관측 위치로 바로 갈 수 있는지"를 planner에 확인한 뒤, 가능하면 anchor를 생략하고 직행**하도록 바꿨습니다.

즉, 안전 fallback은 유지하면서도, 사람이 보기에는 더 직선 위주로 바로 움직이도록 개선한 작업입니다.

---

### 배경

문제의 핵심은 Nav2가 길을 이상하게 만든 것이 아니라, `robot_manual_command_executor.py`가 애초에 `route_anchor -> final_observation` 구조를 먼저 강제하고 있었다는 점이었습니다.

특히 23번 토마토 사례에서는 아래 흐름이 확인됐습니다.

- 프론트엔드는 최종 목표를 `x=1.85, y=6.0`으로 보냄
- 하지만 runtime command status는 `navigationPhase=route_anchor`
- 실제 targetPose도 `x=0.0, y=6.0`인 중간 anchor로 바뀜
- 따라서 로봇은 처음부터 토마토 방향으로 곧게 가지 않고, 통로 중심 anchor를 먼저 향함

이 방식은 멀리 있는 목표나 복잡한 구간에서는 안전하지만, 이미 최종 목표가 충분히 가깝고 planner도 직행 경로를 만들 수 있는 상황에서는 오히려 불필요한 우회를 만듭니다.

그래서 이번 변경의 목적은 다음과 같습니다.

- 가까운 토마토는 먼저 최종 관측 위치 직행 가능 여부를 확인
- 직행 가능하면 `route_anchor`를 생략
- 직행이 불가능하면 기존 `route_anchor -> final_observation` 구조 그대로 fallback

---

### 주요 변경 사항

1. `ActiveCommandContext`에 "직행 probe를 이미 시도했는지 / 지금 진행 중인지" 상태를 추가했습니다.

- 파일: `S14P21A602/agribot_ws/src/agribot_bringup/agribot_bringup/robot_manual_command_executor.py`
- 필드 추가:
  - `direct_final_observation_probe_attempted`
  - `direct_final_observation_probe_inflight`

이 상태값으로 같은 명령에서 직행 확인을 무한 반복하지 않도록 막았습니다.

2. 가까운 2단계 목표만 직행 probe를 시작하는 판단 함수를 추가했습니다.

- 함수: `should_try_direct_final_observation_path(...)`

이 함수는 아래 조건이 모두 맞을 때만 `True`를 반환합니다.

- 현재 단계가 `ROUTE_ANCHOR`
- route anchor와 final target이 실제로 분리된 2단계 목표
- 아직 직행 probe를 시도하지 않음
- 기능이 활성화돼 있음
- 현재 로봇 위치가 final target과 충분히 가까움

즉, 멀리 있는 목표까지 무조건 직행을 시도하지는 않도록 제한했습니다.

3. 수동 이동 실행기에 "직행 path probe -> 가능하면 직행, 실패하면 anchor fallback" 흐름을 추가했습니다.

- 추가 함수:
  - `_probe_direct_final_observation_path`
  - `_handle_direct_final_observation_probe_goal_response`
  - `_handle_direct_final_observation_probe_result`
  - `_dispatch_route_anchor_after_direct_probe_fallback`

동작 흐름은 아래와 같습니다.

1. 현재 목표가 `route_anchor` 단계인지 확인
2. 현재 위치가 final target과 가깝다면 `ComputePathToPose`로 직행 가능 여부 확인
3. path가 실제로 존재하면 `FINAL_OBSERVATION` 단계로 바로 전환
4. path가 없거나 probe가 실패하면 기존처럼 `route_anchor`를 먼저 보냄

이렇게 하면 23번처럼 가까운 목표에서 "왜 저쪽 통로부터 먼저 가지?" 같은 움직임을 줄일 수 있습니다.

4. 관련 파라미터를 추가해 추후 팀이 쉽게 조정할 수 있도록 했습니다.

- `direct_final_observation_path_probe_enabled`
- `direct_final_observation_path_probe_max_distance_m`

기본값은 `enabled=true`, `max_distance_m=2.8`로 설정했습니다.  
즉, 현재 로봇이 final target과 대략 2.8m 이내일 때만 직행 시도를 하도록 제한했습니다.

5. 회귀 방지를 위해 순수 함수 수준 테스트를 추가했습니다.

- 파일: `S14P21A602/agribot_ws/src/agribot_bringup/test/test_robot_manual_command_executor.py`

추가한 테스트:

- 가까운 2단계 목표일 때 `True`
- 이미 probe를 시도한 경우 `False`
- 멀리 있는 목표일 때 `False`

이 테스트로 "가까운 목표만 직행 시도"라는 규칙이 유지되는지 확인했습니다.

## [S14P-139] [ROS/Nav2] 수동 이동 실행기를 통로 기반 세그먼트 상태기로 리팩토링해 회전-직진 중심 주행 구조로 전환

### 개요

기존 수동 이동 구조는 목표를 더 잘 고르더라도, 실제 실행은 결국 `NavigateToPose` 또는 `NavigateThroughPoses`에 많이 의존하고 있었습니다.

그래서 waypoint를 잘 선택해도 실제 움직임은 여전히

- 곡선처럼 꺾이거나
- 제자리에서 각도를 오래 맞추고
- 통로 중심선 대신 부드러운 보정 경로를 따라가며
- 사람이 기대하는 "직선 -> 제자리 회전 -> 직선" 패턴과 다르게 보이는

문제가 계속 남아 있었습니다.

이번 작업에서는 이 구조를 그대로 튜닝하는 대신, **수동 이동 실행기 자체를 "통로 기반 세그먼트 상태기"로 리팩토링**했습니다.

이제 수동 이동은 먼저 patrol waypoint 그래프를 이용해 route를 만든 뒤,

1. 회전(`Spin`)
2. 직선 주행(`DriveOnHeading`)
3. 다음 회전
4. 다음 직선

순서로 실행하는 세그먼트 계획을 우선 사용합니다.  
즉, "목표를 어디로 보낼지"뿐 아니라 "어떻게 갈지"도 바꾼 작업입니다.

---

### 배경

그동안 확인된 핵심 문제는 아래였습니다.

1. 상위 라우팅은 waypoint 중심으로 계산하고 있었지만,
2. 실제 실행은 Nav2 일반 경로 추종이 맡고 있었고,
3. 그 결과 통로형 환경에서도 상용 로봇청소기/서빙 로봇처럼 보이는 직선 중심 동작이 잘 나오지 않았습니다.

특히 23번 토마토 사례에서는

- 프론트엔드가 의도한 target은 비교적 명확해도
- 실행기는 중간 `route_anchor`를 먼저 밟고
- 그 이후 실제 경로는 다시 Nav2 추종으로 넘겨졌기 때문에
- 결과적으로는 "목표를 잘 골라도, 움직임은 여전히 삐뚤삐뚤"해 보였습니다.

따라서 이번 변경의 목적은 아래 2가지였습니다.

- 통로형 환경에 맞는 `행동 분리형 주행` 구조로 바꾼다.
- 기존 Nav2를 완전히 버리지 않고, 세그먼트 실행 실패 시 `fallback`으로 남겨 둔다.

즉, 공격적인 리팩토링이지만 실패했을 때는 기존 구조가 받쳐 주도록 설계했습니다.

---

### 주요 변경 사항

1. `manual_navigation_routing.py`에 세그먼트 실행 계획 계층을 새로 추가했습니다.

- 파일: `S14P21A602/agribot_ws/src/agribot_bringup/agribot_bringup/manual_navigation_routing.py`

추가한 구조:

- `ManualNavigationBehaviorCommand`
- `ManualNavigationBehaviorPlan`
- `build_manual_navigation_behavior_plan(...)`

이제 기존 `ManualNavigationRoute`가 "지나갈 점 목록"을 계산한다면,  
새 `BehaviorPlan`은 "어떤 순서로 회전하고, 어느 구간을 얼마만큼 직진할지"를 계산합니다.

2. 대각선 이동을 축 정렬 세그먼트로 분해하도록 바꿨습니다.

- 추가 함수:
  - `_axis_aligned_subtargets(...)`
  - `_prune_backtracking_route_prefix(...)`

핵심 의도는 아래와 같습니다.

- 통로형 환경에서는 대각선 질주보다 축 정렬 직선이 더 자연스럽다.
- 작은 위치 오차 때문에 경로가 잠깐 뒤로 갔다가 다시 앞으로 오는 backtracking 점을 만들면, 사람이 보기엔 매우 부자연스럽다.
- 그래서 세그먼트 계획 단계에서부터 대각선은 `x 정렬 -> y 정렬` 식으로 나누고, 초반 역방향 prefix는 제거했다.

예를 들어 23번 토마토 route anchor로 들어가는 초기 구간은 이제

- 한 번에 대각선으로 (0, 6)으로 질러가는 것이 아니라
- 먼저 `x=0` 통로 축으로 정렬
- 그 다음 `y=6`까지 직진

하도록 계획됩니다.

3. `robot_manual_command_executor.py`를 세그먼트 상태기 중심으로 바꿨습니다.

- 파일: `S14P21A602/agribot_ws/src/agribot_bringup/agribot_bringup/robot_manual_command_executor.py`

핵심 변경:

- `Spin`, `DriveOnHeading` action client 추가
- `ActiveCommandContext`에 세그먼트 명령 목록과 현재 인덱스 저장
- `_dispatch_navigation_goal(...)`에서 기존 Nav2 batch goal 전에 behavior plan을 우선 시도
- `_dispatch_next_route_behavior_command(...)`로 세그먼트 하나씩 실행
- `_handle_route_behavior_goal_response(...)`
- `_handle_route_behavior_result(...)`
- `_fallback_after_route_behavior_failure(...)`

즉 실행 흐름이 이제 아래처럼 바뀝니다.

1. manual route 계산
2. behavior plan 계산
3. `spin -> drive_on_heading -> spin -> drive_on_heading ...`
4. 모든 세그먼트 성공 시 다음 phase로 전진
5. 세그먼트 실패 시 기존 Nav2 일반 목표로 fallback

4. 기존 성공 처리 분기를 공통 함수로 묶었습니다.

- 추가 함수:
  - `_complete_navigation_stage_success(...)`

기존에는 `NavigateToPose` 결과 처리 안에서만

- `route_egress -> route_anchor`
- `route_anchor -> final_observation`
- 최종 완료

흐름을 처리하고 있었는데, 이제는 세그먼트 주행 성공 시에도 동일한 후속 분기를 써야 하므로 공통 함수로 정리했습니다.

이 리팩토링으로 실행 방식이 달라도 phase 전환 규칙은 한 곳에서 유지됩니다.

5. 세그먼트 주행의 선점 취소 경로도 별도로 추가했습니다.

- 추가 함수:
  - `_request_active_behavior_cancel_for_preemption(...)`
  - `_handle_active_behavior_cancel_response(...)`

이 부분이 없으면 새 목표, 일시정지, 비상정지 명령이 들어왔을 때

- 회전 행동이 끝날 때까지 대기하거나
- 중간 세그먼트가 남아 있는 상태로 흐름이 꼬일 수 있습니다.

그래서 세그먼트 행동도 기존 navigate goal과 마찬가지로 선점 취소가 가능하도록 맞췄습니다.

6. 관련 파라미터를 추가해 추후 조정 가능하게 했습니다.

- `spin_action`
- `drive_on_heading_action`
- `route_behavior_execution_enabled`
- `route_behavior_drive_speed_mps`
- `route_behavior_final_drive_speed_mps`
- `route_behavior_spin_threshold_rad`

즉 이번 구조 변경은 코드에 하드코딩된 일회성 해킹이 아니라, 이후 팀이 실제 주행 체감에 맞춰 속도나 회전 임계값을 계속 조정할 수 있게 열어 둔 형태입니다.

## [main-주석] [ROS/Nav2] 수확 접근을 직선-회전-직선 중심 staged 경로로 재설계

### 개요

수확 목표를 선택했을 때 agribot이 한 번에 목적지로 부드럽게 들어가지 못하고, 반대 방향으로 잠깐 가거나 곡선으로 삐뚤삐뚤 접근하는 문제가 있었습니다.

이번 작업에서는 수확 경로를 다시 설계해서, 가능한 한 다음 흐름으로 움직이도록 바꿨습니다.

1. 현재 패트롤 통로 기준으로 시작 위치를 잡는다.
2. 다른 통로로 갈 때는 통로 끝단(headland)으로 먼저 이동한다.
3. 옆 통로로 직선 이동한다.
4. 목표 행의 inspect 지점까지 직선 이동한다.
5. 마지막으로 토마토 쪽으로 짧게 직선 접근한다.
6. 필요할 때만 마지막 제자리 정렬을 한다.

즉, 일반적인 서비스 로봇처럼 보이도록 `직선 주행 + 제자리 회전 + 직선 주행` 중심의 staged path를 추가한 작업입니다.

---

### 배경

문제의 핵심은 센서가 부족해서라기보다, 수확 라우팅과 실제 이동 목표를 주는 방식이 “관측 통로 1개를 고른 뒤, 그쪽으로 단일 Nav2 goal을 보내는 구조”였다는 점이었습니다.

특히 아래 조건이 겹치면서 이동이 더 어색해졌습니다.

- 한 토마토를 두 개의 통로가 동시에 관측할 수 있음
- 현재 위치 기준 후보 선택이 `/odom` 값에 의존함
- 실제 목표는 토마토 앞 접근점이 아니라 inspect waypoint 쪽으로 먼저 잡힘
- Nav2는 단일 goal 기준으로 곡선형 접근을 만들 수 있음

그래서 “현재 통로를 따라 끝까지 갔다가 옆 통로로 넘는” 구조가 코드에 명시적으로 들어 있지 않았고, 그 결과 사용자가 보기에는 비정상적으로 꺾이거나 돌아가는 것처럼 보였습니다.

### 주요 변경 사항

1. 수확 라우팅에 staged approach path를 추가했습니다.

- `harvest_routing.py`의 `HarvestRoutePlan`에 `approach_path`를 추가했습니다.
- 이 경로는 단일 목표점이 아니라, 수확 접근 전에 순서대로 밟아야 하는 중간 goal 목록입니다.
- 현재 위치와 목표 inspect waypoint의 x축이 다르면, 두 통로 사이를 바로 가로지르지 않고 headland를 거치도록 계산합니다.
- 같은 통로면 inspect 지점까지 직선으로 가고, 마지막에만 crop 쪽으로 짧게 접근하도록 정리했습니다.

2. 수확 경로를 고를 때 map 기준 시작점을 더 안정적으로 사용하도록 보완했습니다.

- `harvest_route_node.py`, `harvest_action_server.py`에 `route_planning_reference_pose` helper를 넣었습니다.
- patrol status 안에 있는 `current_waypoint_id`, `segment_target_waypoint_id`, `next_waypoint_id`를 우선 사용합니다.

3. `harvest_route_node`가 staged path를 실제로 순서대로 따라가도록 수정했습니다.

- 이제는 `approach_path`를 큐처럼 사용해서, 각 중간 goal을 하나씩 순서대로 보냅니다.

- 기존 fallback인 `inspect waypoint` 재시도는 그대로 유지해서, 명시적으로 그 모드일 때는 예전처럼 안전 관측 지점만 쓰도록 했습니다.

4. `harvest_action_server`도 같은 staged path를 따라가도록 맞췄습니다.

- action server 쪽도 단일 approach pose 대신 `approach_path` 전체를 순차적으로 실행하도록 변경했습니다.
- 즉, UI/미션 브리지 경로와 action 경로가 서로 다른 주행 방식을 쓰지 않도록 정리했습니다.

5. 기본 launch 설정을 staged path 사용 방식에 맞게 바꿨습니다.

- `navigation.launch.py`에서 `harvest_navigation_target_mode` 기본값을 `inspect_waypoint`에서 `approach_pose`로 바꿨습니다.
- 단, 장애나 실패 시 inspect waypoint fallback은 그대로 남겨서 안전망은 유지했습니다.

6. 테스트를 추가해 회귀를 막았습니다.

- 같은 통로에서 `inspect -> approach` 순서가 만들어지는지
- 다른 통로일 때 `headland -> 옆 통로 -> inspect -> approach` 순서가 만들어지는지
- `harvest_route_node`가 staged path의 첫 번째 pose부터 실제로 사용하는지
를 테스트로 검증했습니다.

## [S14P-주석-센서] [ROS/Gazebo] agribot 실내 주행 보강을 위한 자력계·범퍼 센서 추가

### 개요

agribot이 출발 직후 좌우로 두리번거리듯 흔들리며 주행하는 증상을 보고, 원인이 `GPS`나 `지자기 센서` 부재인지부터 다시 점검했습니다.

결론부터 말하면, 현재 증상을 `GPS가 없어서 생긴 문제`로 단정하는 것은 맞지 않았습니다.

- 현재 프로젝트에는 이미 `카메라`, `LiDAR`, `IMU`, `odom`, `TF`, `AMCL/Nav2` 기반 주행 구조가 있습니다.
- 실내 서비스 로봇과 로봇청소기는 보통 `GPS`보다 `LiDAR`, `3D/깊이 센서`, `비전`, `IMU`, `접촉 센서`를 더 핵심적으로 사용합니다.
- 따라서 이번 작업은 `실내 로봇에 더 가까운 센서 구성`을 보강하되, **지금 흔들림의 직접 원인을 해결하는 척하는 무의미한 GPS 추가는 하지 않는 방향**으로 진행했습니다.

그 대신 앞으로 방향 진단과 접촉 기반 복구 로직을 얹을 수 있도록 아래 센서를 추가했습니다.

1. 자력계(`magnetometer`)
2. 앞/좌/우 범퍼 접촉 센서(`contact`)

---

### 배경

사용자는 로보락 로봇청소기나 식당 서빙 로봇처럼 “한 방향을 기준으로 안정적으로 움직이는 센서”가 agribot에도 필요한지 질문했습니다.

이와 관련해 공식 자료를 확인해보면, 실내 로봇은 대체로 아래와 같은 센서 조합을 사용합니다.

- 로보락 공식 자료
  - `StarSight Autonomous System`은 `3D ToF`와 `solid-state LiDAR` 기반 정밀 위치 추정과 장애물 인식을 강조합니다.
  - 즉, 실내 자율주행의 핵심을 GPS가 아니라 `3D 인지 + LiDAR 기반 위치 추정`으로 설명합니다.
- KEENON 공식 자료
  - `LiDAR SLAM + 스테레오 비전`, `VSLAM`, 다중 안전 센서를 통해 실내 배송 로봇이 움직인다고 설명합니다.
  - 역시 실내 서비스 로봇 핵심은 `지도·비전·LiDAR·안전 센서`이지 GPS가 아닙니다.

이 프로젝트도 이미 카메라, LiDAR, IMU가 있는데도 흔들린다는 점을 보면, 주된 원인은 센서 “개수 부족”보다 아래에 가까웠습니다.

- localization/TF 흐름
- Nav2 controller 파라미터
- local/global costmap에서 실시간 장애물 계층을 꺼 둔 상태
- 맵과 실제 센서 프레임 정렬 문제

그래서 이번에는 **상용 실내 로봇에서 흔히 볼 수 있는 방향 보조/접촉 안전 레이어를 추가하되, 현재 증상을 GPS 한 방으로 해결하려는 잘못된 방향은 피하는 것**을 목표로 잡았습니다.

즉, 시뮬레이터 내부에만 센서를 두는 것이 아니라 ROS 2 노드가 바로 구독할 수 있도록 연결까지 마쳤습니다.

## [S14P-주석-GUI] [Gazebo GUI] 공용 시뮬레이터 레이아웃에서 헤더와 패널을 다시 보이게 복구

### 개요

최근 프로젝트 실행 시 Gazebo에서 상단 헤더와 오른쪽 사이드 패널이 사라져, 월드 정보 확인이나 엔티티 선택, 시뮬레이션 제어 버튼 사용이 거의 불가능한 상태였습니다.

확인 결과, 이것은 Gazebo 자체 고장이 아니라 **프로젝트가 기본 GUI 대신 저장소 안의 전용 GUI 레이아웃 파일을 강제로 사용하고 있었고, 그 파일이 패널을 숨기도록 설정되어 있었기 때문**이었습니다.

그래서 이번 작업에서는 로컬 사용자별 Gazebo 기본 설정으로 되돌리는 대신, **프로젝트 공용 GUI 설정 파일 자체를 수정해 팀원이 동일한 UI를 보면서도 필요한 패널을 다시 사용할 수 있도록 복구**했습니다.

---

### 배경

현재 프로젝트의 `spawn_agribot.launch.py`는 Gazebo 실행 시 아래 GUI 설정 파일을 항상 `--gui-config`로 넘기고 있습니다.

- `agribot_ws/src/agribot_description/config/frontend_aligned_gui.config`

즉, 각자 로컬의 `~/.gz/sim/.../gui.config`를 쓰는 것이 아니라 저장소에 있는 공용 레이아웃이 우선합니다.

문제는 이 파일 안에 아래와 같은 숨김 설정이 들어 있었다는 점입니다.

- 메뉴 drawer 숨김
- plugins 메뉴 숨김
- Scene Manager 숨김
- Interactive view control 숨김

이 상태에서는 3D View만 크게 남고, 원래 Gazebo에서 자주 쓰는 헤더/패널이 보이지 않아 디버깅과 조작이 불편했습니다.

---

### 주요 변경 사항

1. 상단 메뉴와 플러그인 목록을 다시 보이게 했습니다.

- `drawer visible="true"`
- `plugins visible="true"`

이 변경으로 Gazebo 기본 UI처럼 상단 메뉴와 플러그인 접근 경로를 다시 사용할 수 있습니다.

2. 오른쪽 정보 패널 역할의 플러그인을 다시 노출했습니다.

- `Scene Manager`
- `Component inspector`
- `Entity tree`

이 패널들은 아래 작업에 직접 도움이 됩니다.

- 월드에 어떤 모델이 올라와 있는지 확인
- 센서나 링크가 실제로 생성됐는지 확인
- 선택한 엔티티의 속성을 GUI에서 바로 확인

3. 자주 쓰는 시뮬레이션 제어 패널을 공용 레이아웃에 추가했습니다.

- `World control`
- `World stats`
- `Interactive view control`

즉, 실행/일시정지/step, 시뮬레이션 시간, 실시간 배율, 뷰 조작 같은 기본 기능을 다시 GUI에서 확인할 수 있도록 했습니다.

---

### 왜 로컬 기본값으로 완전히 되돌리지 않았는가

사용자 로컬의 `~/.gz/sim/.../gui.config`를 그대로 쓰게 만들 수도 있었지만, 그렇게 하면 팀원마다 Gazebo 화면 구성이 달라질 수 있습니다.

- 팀원 모두 같은 레이아웃을 봄
- 로컬 사용자 설정 차이에 덜 영향받음

## [S14P-주석-주행튜닝] [Nav2/AMCL] agribot을 직선 우선·제자리 회전 성향으로 대대적으로 조정

### 개요

agribot이 출발 직후 좌우로 흔들리거나, 목표점을 향해 가면서 곡선으로 계속 비틀리는 문제를 줄이기 위해 주행 설정을 전반적으로 조정했습니다.

이번 작업의 목표는 단순히 “덜 흔들리게” 만드는 것이 아니라, 아래와 같은 이동 성향에 더 가깝게 바꾸는 것이었습니다.

1. 통로에서는 가능한 한 반듯하게 직선으로 이동
2. 방향이 많이 틀어졌다면 움직이면서 얼버무리지 말고 먼저 제자리에서 회전
3. 줄 끝이나 막힌 구간에서는 작은 각도로 꿈틀거리기보다 분명한 회전/후진 복구 수행
4. 벽 따라가기나 탐색 시작 구간도 더 보수적으로 움직여 좌우 출렁임을 줄임

즉, “일반적인 곡선 경로 추종”보다 `직선 우선 + 필요 시 제자리 회전` 쪽으로 성향을 바꾼 작업입니다.

---

### 배경

이전 점검에서 확인한 것처럼, 현재 agribot의 좌우 흔들림을 `지자기 센서 부재`만으로 설명하기는 어렵습니다.

이미 프로젝트에는

- LiDAR
- IMU
- odom / TF
- AMCL
- Nav2

가 들어 있으며, 실제 문제는 아래 요소들의 조합일 가능성이 더 컸습니다.

- Regulated Pure Pursuit controller가 방향 오차를 “제자리 회전”보다 “움직이며 곡선으로 보정”하는 성향
- 목표 yaw 허용 오차가 너무 커서 방향을 정확히 맞추지 않고도 도착 처리되는 점
- AMCL이 시뮬레이션치고는 보수적인 노이즈 가정으로 동작하던 점
- frontier explorer가 앞으로 열린 방향보다 좌우 선택을 넓게 허용해 자주 흔들릴 수 있던 점
- collision monitor stop zone이 너무 좁아 정면 물체를 늦게 보고 급히 비트는 점

그래서 이번에는 센서 추가보다 **기존 주행기·로컬라이제이션·탐색 파라미터를 한 묶음으로 조정**하는 것이 맞다고 판단했습니다.

---

### 주요 변경 사항

1. `nav2_params.yaml`을 직선 주행/제자리 회전 성향으로 조정했습니다.

핵심 변경:

- `goal_checker`를 더 엄격하게 조정
  - `xy_goal_tolerance`, `yaw_goal_tolerance` 축소
  - 방향을 제대로 맞춘 뒤 도착 처리되도록 변경
- `use_rotate_to_heading: true`
  - 이 설정이 가장 중요합니다.
  - 방향 오차가 보이면 움직이며 둥글게 보정하기보다 먼저 제자리에서 헤딩을 맞춘 뒤 출발하도록 유도합니다.
- `desired_linear_vel`, `lookahead_dist`, `max_lookahead_dist` 축소
  - 직선 주행 중 좌우 오버슈트를 줄이기 위한 조정입니다.
- `rotate_to_heading_min_angle` 축소
  - 비교적 작은 방향 오차에서도 제자리 회전을 먼저 하도록 만들었습니다.
- `behavior_server`의 회전 속도도 낮춰
  - 빠르게 홱 도는 느낌보다 분명하지만 덜 급한 제자리 회전을 하게 했습니다.

즉, `곡선으로 비틀며 따라가기`보다 `멈춰서 방향 맞추기 -> 직선 이동`으로 무게중심을 옮겼습니다.

2. `nav2_mapping_params.yaml`도 같은 방향으로 조정했습니다.

맵 생성/탐색 중에도 같은 문제가 반복될 수 있으므로, 매핑용 Nav2 설정도 별도로 조정했습니다.

핵심 변경:

- controller 빈도와 timeout 조정
- `use_rotate_to_heading: true`
- 더 작은 lookahead
- 더 엄격한 goal tolerance
- `use_final_approach_orientation: true`

즉, 단순 주행뿐 아니라 “탐색하면서 이동하는 경우”도 마지막 방향을 흐리게 넘기지 않도록 맞췄습니다.

3. `amcl.yaml`에서 시뮬레이션 기준으로 노이즈 가정을 낮췄습니다.

현재 환경은 실제 현장보다

- 오도메트리
- LiDAR
- 맵 정합

이 더 깨끗한 편이라, AMCL이 과하게 보수적인 노이즈를 가정하면 오히려 위치/자세 추정이 흔들릴 수 있습니다.

그래서 다음을 조정했습니다.

- `alpha1~alpha5` 감소
- `transform_tolerance` 축소
- `update_min_a`, `update_min_d` 축소
- `z_hit` 증가, `z_rand` 감소

즉, “시뮬레이션에서는 조금 더 자신 있게 현재 위치를 믿도록” 만든 조정입니다.

4. `frontier_explorer.yaml`에서 앞으로 가는 성향과 직선성을 더 강하게 만들었습니다.

이 노드는 단순 Nav2 목적지 이동이 아니라, 자동 탐사 시 실제로 어디를 향해 움직일지 결정하는 성격이 강합니다.

여기서 다음을 조정했습니다.

- `heading_window_rad` 축소
  - 좌우로 넓게 둘러보며 방향을 자주 바꾸지 않도록 조정
- `heading_forward_bias_weight` 증가
  - 가능한 한 정면 방향을 더 우선하도록 조정
- `bootstrap_drive_speed_mps` 감소
  - 시작 직후 직진 안정성 강화
- `boundary_follow_linear_speed_mps` 감소
- `boundary_follow_max_angular_speed_rps` 감소
- `boundary_follow_side_gain`, `boundary_follow_diagonal_gain` 감소
  - 벽 따라가기 중 작은 거리 오차에 너무 민감하게 좌우로 흔들리는 문제를 줄이기 위한 조정
- `recovery_default_spin_rad: 1.57`
  - 막혔을 때 애매한 작은 각도 대신, 90도에 가까운 분명한 제자리 회전을 먼저 시도하도록 조정

즉, frontier explorer의 기본 성향도 “자주 흔들며 탐색”보다 “앞으로 반듯하게 가다가 필요할 때만 분명히 회전”으로 바꿨습니다.

5. `collision_monitor_mapping.yaml`의 정면 stop zone을 넓혔습니다.

기존 감시 구역은 너무 좁아서, 차체 정면보다 조금만 옆에 있는 물체는 늦게 감지하고 뒤늦게 각을 틀 수 있었습니다.

그래서 stop zone을 더 넓혀

- 줄 끝
- 구조물 앞
- 정면 장애물 근처

에서 “늦게 발견하고 급히 비트는 움직임”을 줄이도록 조정했습니다.

---

### 기대 효과

이번 튜닝으로 기대하는 변화는 아래와 같습니다.

1. 목적지로 갈 때 출발 직후 방향 오차가 있으면 먼저 제자리 회전

## [S14P-139] [문서화/가독성] 저장소 전반의 코드 파일에 상세 주석을 확장 적용

### 개요

이전 작업에서는 실행 흐름을 처음 따라가게 되는 핵심 파일들에 주석을 집중적으로 보강했다면, 이번 작업은 그 범위를 저장소 전반으로 확장했습니다.

즉, `backend`, `frontend`, `agribot_ws`, `scripts`, `tools` 아래의 실제 코드 파일들을 대상으로 파일 역할 설명과 함수/클래스/주요 처리 단계 설명을 추가해, 특정 핵심 파일만이 아니라 프로젝트 전체를 열어 보더라도 흐름을 따라가기 쉬운 상태를 만드는 데 목적을 두었습니다.

이번 작업은 단순히 파일 맨 위에 한 줄만 넣는 것이 아니라, 언어별로 다음 기준을 나눠 적용했습니다.

- Python: 함수, 클래스, 주요 문장 앞 설명 보강
- TypeScript / TSX / JS: 컴포넌트, 함수, hook 구간, 렌더링 구간 설명 보강
- Shell script: 실행 단계, 함수, 핵심 외부 명령 설명 보강
- YAML / XML / SDF / URDF / SQL / ROS interface 파일: 파일 역할과 읽는 포인트 설명 추가

---

### 배경

핵심 실행 파일에만 주석을 늘린 상태에서도 아래 어려움이 남아 있었습니다.

- 다른 서비스 파일이나 ROS 노드를 열면 다시 설명이 끊겨 흐름이 이어지지 않음
- 테스트 파일, launch 파일, 설정 파일을 볼 때 이 파일이 시스템에서 왜 필요한지 한 번에 드러나지 않음
- 프론트 컴포넌트와 보조 유틸 파일은 여전히 맥락 설명이 부족해, 메인 페이지 밖으로 나가면 이해 속도가 다시 떨어짐

그래서 이번에는 “읽는 사람이 어느 파일을 먼저 열더라도 최소한 파일 역할과 읽는 기준은 파악할 수 있게 만든다”는 방향으로 범위를 넓혔습니다.

---

### 주요 변경 사항

1. 저장소 전반의 코드/설정 파일에 파일 책임 주석을 추가했습니다.

### 개요

`main` 브랜치 기준으로 실제 실행되는 코드 파일들에 한글 설명 주석을 일괄 추가했습니다.

이번 작업의 목적은 “이 파일이 왜 존재하는지, 어디에서 쓰이는지”를 팀원이 파일을 열자마자 이해할 수 있게 만드는 것입니다.

특히 비전공자 팀원이나 오랜만에 코드를 다시 보는 팀원이 파일 상단만 읽어도 아래를 빠르게 파악할 수 있도록 맞췄습니다.

1. 이 파일이 백엔드/프론트엔드/ROS2/IoT 중 어느 영역에 속하는지
2. 이 파일이 실행 진입점인지, API 라우터인지, 테스트인지, 보조 스크립트인지
3. 이 파일의 책임이 “화면 구성”, “로봇 제어”, “센서 처리”, “데이터 변환”, “운영 보조” 중 무엇인지

반대로, 일반적으로 실무에서 코드 설명 주석을 달지 않는 문서형 파일은 이번 작업에서 제외했습니다.

- `*.md`
- `*.json`, `*.yaml`, `*.sql`
- `package.xml`, `setup.cfg`, 이미지/모델 자산
- `.gitignore` 같은 환경/무시 규칙 파일

즉 이번 작업은 **“코드 파일은 쉽게 읽히게, 문서/설정/자산 파일은 불필요하게 건드리지 않게”** 정리한 변경입니다.

## [발표용 UI] 수확 성공 후 FastAPI 로그 대신 실제 토마토 사진을 바로 보여주도록 프론트 보정

### 개요

발표 시연에서 `수확하기`가 성공한 뒤, 화면에 백엔드 access log 성격의 긴 문자열이 그대로 노출되던 문제를 정리했습니다.

이번 작업의 핵심은 다음 3가지입니다.

1. 수확 화면이 `harvests/stats`만 보는 구조에서 한 단계 더 나아가, `plants`의 실제 최신 이미지 URL도 함께 읽도록 확장
2. 마지막 수확 대상 fruit ID와 plant ID를 기준으로 실제 토마토 사진을 찾아 수확 화면 상단에 바로 노출
3. `detail_message`에 FastAPI access log가 섞여 들어오더라도 화면 문구로 그대로 출력하지 않도록 프론트에서 안전하게 필터링

즉, 발표용 눈속임을 위해 가짜 이미지를 새로 만드는 대신, **이미 백엔드에 저장되어 있는 실제 토마토 사진 데이터를 우선 사용하도록** 프론트 흐름을 보정했습니다.

---

### 배경

기존 수확 화면은 `harvests/stats`의 `detail_message`를 그대로 보여주는 방향이었습니다.

그런데 실제 시연 환경에서는 아래와 같은 문제가 있었습니다.

- 수확 성공 직후 사용자에게 보여줘야 할 것은 “방금 수확한 토마토의 실제 사진”인데, 화면에는 HTTP access log 문자열이 길게 보임
- 이 문자열은 발표 관점에서 의미 전달력이 낮고, 비전공 팀원이나 청중 입장에서는 왜 저 문장이 뜨는지 이해하기 어려움
- 반면 실제 토마토 사진은 이미 백엔드 `plants` 계층의 `latest_image_url`에 존재할 가능성이 높았음

그래서 이번 작업에서는 **로그 문자열을 UI에 그대로 노출하는 흐름을 끊고, 실제 이미지 데이터 중심으로 수확 성공 결과를 보여주는 방향**으로 정리했습니다.

---

### 주요 변경 사항

1. **수확 페이지 데이터 모델에 “최근 수확 이미지” 정보를 추가했습니다.**
   - `frontend/src/lib/api/agribot.ts`의 `HarvestPageData`에 아래 필드를 확장했습니다.
   - `latestHarvestImageUrl`
   - `latestHarvestImageLabel`
   - `latestHarvestPlantName`
   - `latestHarvestObservedAt`

이렇게 해 두면 수확 페이지는 단순 문자열이 아니라, “최근 수확 대상의 실제 미디어 정보”까지 한 번에 다룰 수 있습니다.

2. **`getHarvestPageData()`가 `plants` 데이터까지 같이 조회하도록 확장했습니다.**
   - 기존에는 `harvests/stats`, `harvests`만 함께 읽었습니다.
   - 이제 `plants`도 같이 읽은 뒤, 마지막 수확 fruit ID와 현재 active target ID, 배치 정보의 plant/fruit ID를 기준으로 일치하는 작물을 찾습니다.
   - 그 작물의 `latest_image_url`과 `latest_display_label`을 수확 화면용 미리보기 데이터로 변환하도록 했습니다.

이 변경 덕분에 발표 중 수확 성공 직후, 프론트가 이미 백엔드에 저장된 실제 토마토 사진을 다시 활용할 수 있게 됐습니다.

3. **FastAPI access log 형태의 문자열은 화면 표시용 문구에서 제외했습니다.**
   - `agribot.ts`에 access log 패턴을 판별하는 보조 함수를 추가했습니다.
   - `detail_message`, `message`, `operator_message`를 읽을 때, `INFO: 127.0.0.1 ... 200 OK` 같은 로그 묶음은 사용자 노출용 텍스트로 채택하지 않도록 했습니다.

이 작업은 발표용 임시 처리이면서도, 장기적으로도 맞는 방향입니다.
UI는 운영자에게 의미 있는 상태만 보여줘야 하고, 서버 access log는 운영 콘솔이나 로그 수집 계층에서 다뤄야 하기 때문입니다.

4. **수확 화면 상단(hero)에 실제 토마토 사진 카드가 보이도록 UI를 보강했습니다.**
   - `frontend/src/pages/harvest-page.tsx`에서 `MockupImage`를 이용해 최근 수확 대상 사진을 상단에 배치했습니다.
   - 사진 라벨에는 백엔드가 가진 최신 진단/관측 라벨을 우선 사용하고, 사진 아래에는 식물명과 마지막 관측 시각을 함께 보여주도록 했습니다.

이제 수확 성공 뒤에는 “문자 로그”가 아니라 “실제 토마토 사진 + 식물 정보”가 먼저 눈에 들어오도록 바뀌었습니다.

## [S14P-138] [ROS/주행] 그래프 기반 최적 모드에서 직선 세그먼트 상태기를 강화해 마지막 접근 흔들림을 줄임

### 개요

그래프 기반 최적 모드를 유지하면서도, 실제 실행이 다시 일반 Nav2 곡선 추종으로 무너지지 않도록 수동 주행 실행기를 대대적으로 정리했습니다.

이번 작업의 핵심은 다음 4가지입니다.

1. 최종 접근 probe가 여러 대체 후보를 기본으로 탐색하던 구조를 바꿔, 기본값은 `final target 1개만 확인`하도록 단순화
2. `spin / drive_on_heading` 세그먼트가 실패했을 때 곧바로 일반 Nav2로 fallback 하지 않고, 현재 pose 기준 `남은 각도 / 남은 거리`를 다시 계산해 같은 세그먼트를 1회 재시도하도록 변경
3. route pose와 spin 목표 yaw를 동서남북 축 기준으로 더 강하게 정규화해, 마지막 5~15cm 사선 보정 때문에 “두리번거리는 것처럼 보이는” 구간을 줄임
4. 세그먼트가 이미 끝점 근처까지 도달했다면 그 구간을 성공으로 간주하고 다음 세그먼트로 넘기도록 해, 불필요한 곡선형 보정과 과도한 fallback을 줄임

즉, 이번 작업은 “어느 통로를 고를지”보다 **“선택한 통로를 끝까지 직선 세그먼트 + 90도 회전 위주로 밀어붙이는 실행 구조”**를 만드는 데 집중했습니다.

---

### 배경

이전 단계까지는 프론트엔드와 그래프 라우팅 쪽에서 `가까워 보이는 후보` 대신 `실제로 덜 돌아가는 후보`를 고르도록 개선했지만, 실제 Gazebo 주행은 여전히 아래 문제를 보였습니다.

- 첫 직선은 잘 가더라도 마지막 토마토 접근 구간에서 다시 좌우로 흔들림
- `spin -> drive_on_heading` 세그먼트 실행 중 작은 실패가 나면 곧바로 일반 Nav2로 fallback 하면서 곡선 보정이 다시 나타남
- 최종 접근 경로 확인 시 final target 외의 대체 probe target까지 기본으로 함께 검사해, 사람 눈에는 “탐색”처럼 보이는 상태가 생김
- 마지막 raw target pose가 통로 축과 거의 일치해도 그대로 붙으면서 아주 짧은 사선 보정이 남음

즉, **그래프 기반 최적 후보 선택은 좋아졌지만, 실행 계층은 아직 완전한 통로 상태기가 아니어서 마지막 접근 품질이 깨지는 문제**가 남아 있었습니다.

---

### 주요 변경 사항

1. `manual_navigation_routing.py`에서 route pose와 spin 목표를 더 강하게 정규화했습니다.

- 마지막 raw target pose를 그대로 route 끝에 붙이지 않고, 현재 통로 축 기준으로 한 번 더 정규화한 뒤 추가하도록 변경
- `spin` 명령의 `target_pose.yaw`에 실제 목표 헤딩을 저장하도록 바꿔, 재시도 시 “처음 계획한 회전량”이 아니라 “지금 기준 남은 회전량”을 다시 계산할 수 있게 변경
- 대각선 구간은 먼저 `가로/세로 세그먼트`로 분해한 뒤 각 세그먼트만 축 정렬 정규화하도록 바꿔, 최종 x/y 이동 자체가 사라지는 부작용을 막음

이 변경으로 세그먼트 실행기가 항상 “동서남북 축 기준의 긴 직선 + 제자리 회전”에 더 가까운 명령을 받도록 정리했습니다.

2. `robot_manual_command_executor.py`에서 세그먼트 실패 처리 전략을 바꿨습니다.

- `route_behavior_retry_limit`, `route_behavior_soft_complete_xy_tolerance_m`, `route_behavior_nav2_fallback_enabled`, `final_observation_intermediate_probe_enabled` 파라미터 추가
- 기본값은
  - `세그먼트 1회 재시도`
  - `세그먼트 끝점 근접 성공 허용`
  - `일반 Nav2 fallback 비활성화`
  - `최종 접근 대체 probe 비활성화`
  로 두었습니다.

이 기본값은 “성공률만 높이기 위해 곡선형 fallback을 남발하지 않고, 직선 세그먼트 상태기를 우선한다”는 현재 프로젝트 목적에 맞춘 것입니다.

3. 최종 접근 probe 기본 전략을 바꿨습니다.

- 이전: `final target + intermediate 대체 후보들`을 기본으로 모두 검사
- 현재: `final target`만 기본으로 검사
- 필요할 때만 파라미터로 intermediate probe를 다시 켤 수 있도록 분리

이 변경으로 마지막 접근 직전, 로봇이 여러 대체 점을 “여기저기 생각해 보는 것처럼 보이는” 상태를 기본 경로에서 제거했습니다.

4. 세그먼트 실패 시 곧바로 Nav2로 복귀하지 않도록 상태기를 강화했습니다.

- `drive` 실패 시 현재 위치에서 목표 세그먼트 끝점까지의 `남은 거리`를 다시 계산해 같은 직선을 재시도
- `spin` 실패 시 현재 yaw와 목표 yaw의 차이만 다시 계산해 남은 각도만 재시도
- 이미 허용 오차 안에 들어온 세그먼트는 성공 처리 후 다음 세그먼트로 진행
- 재시도 한도를 넘기면, 기본 정책에서는 일반 Nav2로 내려가지 않고 명령을 실패로 종료
- 팀이 필요하면 파라미터로 `Nav2 fallback`을 다시 켤 수 있게 옵션은 남겨 둠

이 구조 덕분에 “세그먼트 상태기 -> 작은 실패 -> 즉시 일반 Nav2 곡선 추종”으로 흐름이 깨지는 문제가 줄어듭니다.

5. 회귀 테스트를 보강했습니다.

- 최종 접근 probe 기본 전략이 `final target only`인지 검증
- 세그먼트 재시도 여부가 retry limit을 지키는지 검증
- `drive` 재시도 시 남은 거리만 계산하는지 검증
- `spin` 재시도 시 남은 회전각만 계산하는지 검증
- 마지막 route pose 축 정렬 정규화가 기대대로 동작하는지 검증

즉, 이번 리팩토링은 단순 구현 변경이 아니라, **상태기 설계를 테스트로 함께 잠가 두는 방향**으로 정리했습니다.

## [main-주석] [CLI/개발환경] Codex CLI·Gemini CLI 완료 알림을 부모 셸 래퍼 방식으로 전면 재구성

### 개요

안티그래비티 터미널에서 `codex cli`와 `gemini cli`의 답변 완료 시점에 상단 알림이 오도록 만드는 작업을, 기존의 꼬여 있던 훅/내장 옵션 기반 설정에서 **부모 셸 래퍼 기반 구조**로 전면 재구성했습니다.

이번 작업의 핵심은 다음 4가지입니다.

1. 왜 예전에 한 번은 보였고, 그 뒤에는 안 보였는지 실제 원인을 CLI 구현과 실행 환경 기준으로 분리 분석
2. Ubuntu에서 실질적으로 무효한 Gemini 내장 알림과, TTY 없이 실행되는 Codex notify 훅 의존 구조 제거
3. 사용자가 실제로 명령을 실행한 `bash` 부모 셸이 종료 직후 직접 `OSC 9` 알림을 보내는 공통 래퍼 도입
4. 저장소 안 스크립트 하나로 `~/.bashrc`, `~/.codex/config.toml`, `~/.config/gemini/settings.json`을 다시 맞추는 설치 방식으로 정리

즉, 이번 작업은 “한 번은 되던 우연한 알림”을 유지하는 수준이 아니라, **왜 실패했는지 설명 가능한 구조로 바꾸고, 이후에도 다시 설치해서 같은 결과를 재현할 수 있게 만든 정리 작업**입니다.

---

### 배경

기존에는 알림 관련 설정이 아래처럼 섞여 있었습니다.

- Gemini: `enableNotifications`, `AfterAgent` 훅, `/dev/tty` 직접 출력 시도 등이 시기별로 섞여 있었음
- Codex: `notify = [...]` 훅, `[tui] notification_method = "osc9"`, 별도 hook 스크립트가 동시에 존재했음
- 홈 디렉터리 안에 `notify.sh`, `sound.sh` 같은 실험용 파일이 남아 있어 실제 동작 경로를 추적하기 어려웠음

문제를 정확히 확인하기 위해 실제 설치된 CLI 코드와 실행 로그를 확인했습니다.

1. Gemini CLI 원인

- 현재 설치된 `gemini-cli 0.35.3` 구현을 확인한 결과, `general.enableNotifications`는 사실상 `process.platform === 'darwin'`일 때만 동작하도록 되어 있었음
- 즉, 현재 Ubuntu 환경에서는 이 옵션을 켜도 상단 알림이 오지 않는 구조였음

2. Codex CLI 원인

- 현재 설치된 `codex-cli 0.117.0`는 `notify` 훅과 `osc9` 관련 코드를 모두 가지고 있었음
- 하지만 실제 `notify` 훅 실행 환경을 로그로 찍어 보니 `TTY=not a tty` 상태로 실행됨
- 그래서 훅 안에서 `/dev/tty`로 직접 쓰는 방식은 구조적으로 실패했고, 훅의 stdout/stderr도 사용자 터미널에 그대로 보이지 않아 `OSC 9` 시퀀스가 화면까지 전달되지 않았음

3. “예전에는 한 번 보였던 이유”

- 사용자가 예전에 봤던 상단 메시지는 CLI 내부 훅이 성공해서가 아니라, **부모 셸 또는 직접 실행한 shell 명령이 터미널에 `OSC 9`를 바로 출력했을 때만 우연히 보였던 것**으로 정리됨
- 즉, 알림을 안정적으로 보내려면 “CLI 내부 하위 프로세스”가 아니라 “사용자가 직접 실행한 부모 셸”이 알림을 쏘도록 바꿔야 했음

### 주요 변경 사항

1. 부모 셸 기준의 공통 알림 스크립트를 새로 만들었습니다.

- `scripts/emit_cli_notification.sh`

이 스크립트는 다음 역할을 담당합니다.

- `OSC 9` 시퀀스를 직접 출력해 안티그래비티/VS Code 계열 통합 터미널 상단 알림을 유도
- 메시지에 개행, `BEL`, `ESC` 같은 제어 문자가 섞여 알림이 깨지지 않도록 정리
- 가능하면 시스템 오디오 재생을 시도하고, 안 되면 터미널 bell로 fallback
- 필요 시 `CLI_NOTIFY_SYSTEM=1`일 때만 `notify-send`를 보조 수단으로 사용

즉, 알림을 “어디서 보낼지”를 한 곳으로 모아, 이후 조정도 이 파일만 보면 되도록 만들었습니다.

2. `codex`, `gemini`를 감싸는 bash 래퍼를 새로 만들었습니다.

- `scripts/cli_notify_shell.sh`

이 파일은 interactive bash에서 다음처럼 동작합니다.

- 사용자가 `codex ...`, `gemini ...`를 실행
- 실제 명령은 그대로 실행
- 명령이 끝난 뒤 부모 셸이 종료 코드와 실행 시간을 계산
- 2초 이상 걸린 실행에 대해서만 공통 알림 스크립트를 호출

핵심은 **알림을 하위 훅 프로세스가 아니라, 사용자가 실제로 붙어 있는 부모 셸이 보낸다**는 점입니다.

이 구조 덕분에 TTY가 없는 내부 훅 문제를 우회할 수 있습니다.

3. 설치 스크립트로 홈 디렉터리 설정 정리 작업을 자동화했습니다.

- `scripts/install_cli_notifications.sh`

이 스크립트는 한 번 실행하면 아래를 정리합니다.

- `~/.bashrc`에 저장소 래퍼 파일 source 블록을 idempotent하게 추가
- `~/.codex/config.toml`에서 기존 `notify = ...`, `[tui] notifications = ...`, `notification_method = "osc9"` 제거
- `~/.config/gemini/settings.json`에서 Ubuntu에서 무효한 `enableNotifications`와 예전 알림 훅 제거
- `~/.codex/hooks/notify.sh`, `~/.codex/hooks/sound.sh`, `~/.config/gemini/hooks/notify.sh` 같은 과거 실험 파일 제거

즉, “현재 한 번 고친 상태”가 아니라, **나중에 다시 꼬여도 같은 스크립트로 재설치해서 복구할 수 있는 구조**를 만든 것입니다.

4. 현재 세션 기준 후속 사용 방법도 명확하게 정리했습니다.

- `.bashrc`를 수정했으므로 현재 열려 있는 터미널에는 즉시 반영되지 않음
- 사용자는 `source ~/.bashrc` 또는 안티그래비티 터미널 재시작이 필요
- 그 이후부터 새로 실행하는 `codex`, `gemini` 세션은 부모 셸 래퍼를 통해 알림을 보냄

즉, “지금 실행 중인 이미 열린 세션”과 “이후 새로 시작할 세션”의 적용 범위를 혼동하지 않도록 정리했습니다.

### 후속 보정

직후 추가 검증에서, 사용자가 말한 “`colcon build`는 우분투 상단 알림이 잘 오는데 `codex/gemini`만 안 온다”는 차이도 반영해 후속 보정을 넣었습니다.

- 기존 기본 동작은 `OSC 9` 중심이었는데, 이는 터미널이 해당 시퀀스를 지원하지 않으면 상단 메시지가 뜨지 않을 수 있음
- 반면 `colcon build` 쪽은 실제 데스크톱 세션의 `notify-send` 경로가 살아 있다는 뜻이므로, `codex/gemini`도 같은 경로를 기본으로 쓰는 편이 더 일관됨

그래서 후속 수정으로 아래를 반영했습니다.

- `emit_cli_notification.sh` 기본 전략을 `notify-send 자동 사용 + OSC 9 보조` 방식으로 조정
- `CLI_NOTIFY_LOG_FILE` 디버그 로그를 넣어, 실제 사용자 터미널에서 어떤 분기가 실행됐는지 추적 가능하게 함
- `cli_notify_shell.sh`의 중복 로드 방지 플래그를 export 하지 않도록 수정해, `source ~/.bashrc` 또는 `exec bash` 이후 자식 셸에서도 함수 정의가 사라지지 않게 보정

후속 테스트 로그에서는 다음이 확인됐습니다.

- `system-notify=ok`
- 즉, 현재 보이는 세션 기준 환경 변수(`DISPLAY`, `DBUS_SESSION_BUS_ADDRESS`)로는 `notify-send` 경로가 정상 동작함

따라서 이후에는 “Codex/Gemini만 안 오는 구조”를 줄이고, 일반 Ubuntu 상단 알림 흐름에 더 가깝게 맞출 수 있습니다.

### 2차 후속 보정

실사용 확인 단계에서, `codex exec`처럼 한 번 실행하고 끝나는 경우는 알림이 왔지만, 사용자가 실제로 원한 **대화형 세션에서 매 답변(turn) 완료 시 알림**과는 기준이 다르다는 점을 다시 확인했습니다.

그래서 알림 전략을 한 번 더 바꿨습니다.

1. process-exit 래퍼를 기본 경로에서 제외했습니다.

- 기존 `~/.bashrc` source 래퍼는 “명령이 완전히 종료될 때”만 알림을 보내므로
- `gemini -p` 같은 단발성 실행에는 맞지만
- 계속 이어지는 대화형 세션에서는 “답변이 끝나도 프로세스가 안 끝나서 알림이 안 오는” 문제가 있었음

그래서 install 스크립트가 이제는 `.bashrc`에 넣었던 이전 래퍼 블록을 제거하도록 바꿨습니다.

2. Codex는 turn 완료용 내장 notify 훅으로 전환했습니다.

- OpenAI Codex config reference 기준으로 `notify`는 “agent finishes a turn” 시점에 실행됩니다.
- 그래서 `~/.codex/config.toml`에 아래 형태로 turn 완료 훅을 다시 심었습니다.

```toml
notify = ["/home/ssafy/SSAFY/S14P21A602/scripts/cli_turn_notify.sh", "Codex CLI"]
```

- 공통 스크립트는 시스템 알림(`notify-send`)을 우선 사용하고, 필요하면 소리 재생도 시도합니다.

직접 검증 결과, Codex는 turn 훅 로그에 아래처럼 남았습니다.

- `system-notify=ok title=Codex CLI body=답변 완료 | ...`

즉, Codex는 이제 `한 번의 답변이 끝날 때마다` 알림을 보내는 경로로 바뀌었습니다.

3. Gemini는 실제 사용 중인 전역 설정 파일 경로를 바로잡아 AfterAgent 훅을 심었습니다.

처음에는 `~/.config/gemini/settings.json`만 수정했는데, 번들 코드와 실제 설치 경로를 다시 확인해 보니 현재 사용 중인 전역 설정은 `~/.gemini/settings.json` 쪽이었습니다.

즉, 훅이 안 돌던 직접적인 원인은 **설정 파일 경로를 잘못 잡은 것**이었습니다.

그래서 install 스크립트를 보강해 아래 두 경로를 함께 정리하도록 바꿨습니다.

- `~/.gemini/settings.json`
- `~/.config/gemini/settings.json`

실제 사용 경로인 `~/.gemini/settings.json`에는 아래 훅이 들어가도록 맞췄습니다.

```json
"hooks": {
  "AfterAgent": [
    {
      "matcher": "*",
      "hooks": [
        {
          "name": "ssafy-turn-notify",
          "type": "command",
          "command": "/home/ssafy/SSAFY/S14P21A602/scripts/cli_turn_notify.sh \"Gemini CLI\""
        }
      ]
    }
  ]
}
```

직접 재검증 결과, Gemini도 turn 훅 로그에 아래처럼 남았습니다.

- `system-notify=ok title=Gemini CLI body=답변 완료 | ...`

즉, Gemini 역시 이제는 process 종료가 아니라 `AfterAgent`, 즉 답변 완료 시점에 알림이 오도록 정리됐습니다.

4. 공통 turn 알림 진입점도 별도 스크립트로 분리했습니다.

- `scripts/cli_turn_notify.sh`

이 파일은 Codex와 Gemini가 서로 다른 방식으로 훅을 호출하더라도,

- “이건 turn 완료 알림이다”라는 컨텍스트를 공통 실행기에 전달하고
- 실제 알림 전송은 `emit_cli_notification.sh` 한 곳에서 처리하게 만드는 역할을 합니다.

이렇게 분리해 두면 나중에 제목/본문/소리/로그 방식을 바꿔도 훅 설정 문자열 전체를 다시 뜯어고치지 않아도 됩니다.

### 한 줄 회고

이번 작업은 단순히 알림을 “억지로 한 번 띄우는” 시도가 아니라, **CLI 내부 훅의 구조적 한계를 인정하고 부모 셸 래퍼로 책임을 옮겨, 실제 사용자 터미널에서 재현 가능한 알림 경로를 다시 설계한 정리 작업**입니다.
## [S14P-139] [ROS/주행] 최종 작물 접근 실패를 포기 성공으로 처리하지 않도록 고치고 다른 관측 후보로 재계획하게 개선

### 개요

그래프 기반 최적 모드를 유지하되, 실제 주행에서 문제가 드러난 두 가지를 바로잡았습니다.

1. 최종 작물 접근이 막혔는데도 `goal succeeded`처럼 보이게 처리되던 `anchor_only 성공` 흐름 제거
2. 한쪽 관측 통로가 막혔을 때 다른 관측 후보로 갈아타지 못하던 구조 보완

즉, 이번 작업은 “직선 세그먼트를 더 강하게 고집한다”가 아니라, **실제로 접근 가능한 쪽으로 재계획하고, 접근 실패는 실패로 명확하게 드러나게 만드는 안전한 하이브리드 구조**로 정리한 것입니다.

---

### 배경

최근 그래프 기반 최적 모드와 세그먼트 상태기 리팩토링을 통해 `직선 주행 + 90도 회전`을 더 강하게 만들려 했지만, 실제 Gazebo 시연에서 아래 문제가 확인됐습니다.

- 23번 토마토로 간 뒤, 토마토 쪽으로 90도 꺾은 이후 더 접근하지 못함
- 그런데도 코드상 `안전 관측 경유점까지는 도착했고 ... 접근을 마쳤습니다`처럼 성공으로 처리되는 경우가 있음
- 23번에서 21번으로 이동할 때도 세그먼트 상태기 실패로 명령이 중단됨
- 즉 “반듯한 움직임”을 너무 강하게 우선하다가, 실제 접근 가능성이나 다른 관측 후보 전환보다 상태기 고집이 앞서는 문제가 생김

이건 사용자 관점에서 매우 큰 문제입니다.  
**접근을 못 했으면 실패 또는 재계획이어야 하지, 성공처럼 보이면 안 됩니다.**

그래서 이번에는 마지막 접근만큼은 다시 Nav2에 맡기고, 한쪽 후보가 막히면 다른 후보로 갈아타는 쪽으로 전략을 바꿨습니다.

---

### 주요 변경 사항

1. 최종 접근은 segmented route 강제를 포기하고 Nav2 직접 접근으로 바꿨습니다.

- `route_egress`, `route_anchor` 같은 긴 통로 이동은 기존 그래프/상태기 구조를 유지
- `final_observation` 단계는 `NavigateToPose`로 직접 접근

이유는 명확합니다.

- 긴 통로 이동은 그래프 기반 직선/회전 구조가 잘 맞음
- 하지만 마지막 crop-side 접근은 울타리, costmap, 로컬 회피 영향이 커서
  직선 상태기를 강제하면 오히려 실패가 늘어남

즉, 현재 프로젝트에서는  
**“긴 통로 이동은 구조화, 마지막 근접 접근은 Nav2”**가 더 현실적이라고 판단했습니다.

2. 관측 후보 fallback 구조를 추가했습니다.

- 이미 시도한 관측 waypoint를 `attempted_observation_waypoint_ids`로 기록
- 한쪽 관측 통로에서 최종 접근이 막히면,
  남은 관측 후보 중 그래프 비용이 가장 작은 waypoint를 다시 선택
- 그 후보로 `route_target_pose / final_target_pose / navigation_phase`를 다시 구성하고 주행을 이어감

즉, 한쪽에서 막혔다고 곧바로 포기하지 않고,
**다른 관측 후보가 남아 있으면 그쪽으로 재계획**하도록 바꿨습니다.

3. `anchor_only success`를 제거했습니다.

기존에는 최종 접근이 안 돼도 route anchor 근처에만 와 있으면
`수ucceeded`로 처리되는 코드가 있었습니다.

이번에는 이 흐름을 바꿨습니다.

- 다른 관측 후보가 있으면 -> 그 후보로 재계획
- 다른 관측 후보도 없으면 -> 최종 접근 실패를 실패로 남김

즉,  
**“최종 작물 앞까지 못 갔는데 성공처럼 보이는 상태”를 없앴습니다.**

4. 관련 helper와 테스트를 추가했습니다.

- 시도한 관측 waypoint 이력을 중복 없이 관리하는 helper 추가
- 이미 시도한 후보를 제외하고 다음 관측 waypoint를 고르는 helper 추가
- 테스트에서도
  - 시도 이력 중복 방지
  - 제외 후보를 빼고 다음 후보를 올바르게 고르는지
  를 검증하도록 보강

## [main-주석] [ROS/주행] 최종 토마토 접근을 직선 세그먼트 재시도 중심으로 다시 정리

### 개요

이번 작업은 `토마토 근처 마지막 1m 접근`에서 로봇이 계속 좌우로 까딱거리거나,
충분히 가까이 갔는데도 `goal succeed`가 늦게 뜨는 문제를 줄이기 위해 진행했습니다.

핵심 목표는 아래 3가지였습니다.

1. 마지막 접근도 `직진 + 90도 회전` 느낌을 최대한 유지할 것
2. 마지막 접근이 잠깐 흔들렸다고 바로 다른 전략이나 일반 Nav2 곡선 추종으로 무너지지 않을 것
3. 토마토 앞에 충분히 가까이 갔으면 `도착 성공`으로 판정할 것

즉, 장거리 통로 이동과 마지막 토마토 접근을 같은 방식으로 보지 않고,
`마지막 접근은 더 짧고 더 엄격하게 관리되는 별도 정책`으로 다루도록 executor를 보강했습니다.

### 배경

이전 구조에서는 마지막 토마토 접근을 다시 일반 `NavigateToPose`로 보내는 흐름이 남아 있었습니다.
이 경우 실제 화면에서는 다음 문제가 반복됐습니다.

- 통로까지는 비교적 반듯하게 오는데, 마지막 1m에서 다시 곡선형 보정이 시작됨
- 90도로 꺾은 뒤 한 번에 직진하지 못하고 좌우로 계속 흔들림
- 충분히 가까이 접근했는데도 success가 바로 뜨지 않음
- 세그먼트 주행이 잠깐 실패하면 곧바로 다른 전략으로 무너져 사용자가 보기에 불안정함

특히 이 프로젝트는 농장 통로형 구조라서,
`긴 구간은 통로 중심선 기반`, `마지막 구간은 짧고 단순한 접근`으로 나누는 편이 더 자연스럽습니다.

### 주요 변경 사항

1. 마지막 토마토 접근도 다시 세그먼트 주행 방식으로 통일했습니다.

- `robot_manual_command_executor.py`의 `_dispatch_final_observation_goal(...)`를 수정했습니다.
- 이전처럼 마지막 접근만 일반 Nav2 직접 접근으로 보내지 않고,
  다시 `segmented route`를 사용하도록 변경했습니다.
- 결과적으로 마지막 구간도 `spin -> drive_on_heading` 기반으로 처리됩니다.

2. 마지막 접근 전용 재시도 정책을 추가했습니다.

- `ActiveCommandContext`에 `final_observation_behavior_retry_count`를 추가했습니다.
- 마지막 접근 세그먼트가 잠깐 실패했을 때,
  바로 일반 Nav2 fallback으로 떨어지지 않고 같은 직선 접근을 한 번 더 시도합니다.
- 이 재시도 횟수는 파라미터 `final_observation_behavior_retry_limit`로 관리합니다.

3. 마지막 접근 전용 soft success 기준을 더 현실적으로 조정했습니다.

- `final_observation_soft_complete_xy_tolerance_m` 기본값을 `0.55`로 조정했습니다.
- 따라서 목표 pose의 yaw를 끝까지 다시 맞추지 않아도,
  토마토 앞에 충분히 가까워졌다면 성공으로 처리할 수 있습니다.
- 사용자가 보기에는 `도착 후 더 이상 제자리에서 고개를 돌리지 않고 성공 처리`되는 방향입니다.

4. 세그먼트 실패 후 처리 순서를 더 보수적으로 바꿨습니다.

마지막 접근에서 세그먼트가 실패하면 이제 아래 순서로 처리합니다.

1. 현재 위치가 최종 접근 허용 오차 안이면 성공 처리
2. 아니면 같은 직선 접근을 1회 재시도
3. 그래도 안 되면 다른 관측 후보로 전환
4. 그것도 안 되면 최종 실패

즉, 이전처럼 `실패 -> 즉시 일반 Nav2 fallback` 구조가 아니라,
`실패 -> 근접 성공 확인 -> 재시도 -> 후보 전환` 구조로 바꿨습니다.

5. 로그 문구도 실제 전략에 맞게 수정했습니다.

- 이전에는 모든 세그먼트 실패 로그가 `일반 Nav2 목표로 되돌립니다`처럼 보였는데,
  이제는 `후속 실패 처리로 전환합니다`처럼 더 중립적인 문구로 바꿨습니다.
- 실제 실행 전략과 로그가 어긋나지 않도록 정리한 것입니다.

### 기대 효과

- 마지막 토마토 접근에서도 `한 번 회전 -> 짧게 직진` 패턴이 더 잘 유지됩니다.
- 충분히 가까이 갔는데도 success가 늦게 뜨는 문제가 줄어듭니다.
- 작은 흔들림이나 일시적인 실패 때문에 바로 전체 명령이 무너지는 현상이 줄어듭니다.
- 다른 관측 후보가 남아 있으면 그쪽으로 재계획하는 흐름도 그대로 유지됩니다.

## [S14P-주행정책] [ROS/Nav] 정책형 통로 경로로 재구성해 직선·직각 회전 패턴을 더 강하게 고정

### 개요

이번 작업은 `토마토 근처에 갔을 때 계속 좌우로 까닥거리거나`, `안전 통로 복귀를 반복하면서 goal succeed가 늦게 뜨는 문제`를 줄이기 위해,
수동 이동 경로를 다시 `정책형 통로 주행` 기준으로 재구성한 작업입니다.

핵심은 두 가지입니다.

1. waypoint 그래프를 그대로 최단거리로만 쓰지 않고, `같은 통로면 그대로 직진`, `다른 통로면 북/남 connector를 거쳐 통로를 갈아타는` 정책형 경로를 먼저 만들도록 변경
2. 식물 이동에서 `선택된 inspect waypoint` 자체를 실제 최종 목표로 고정해서, waypoint 도착 후 다시 raw crop-side pose로 한 번 더 파고들며 흔들리던 마지막 접근을 제거

즉, 이번 작업은 "짧아 보이는 경로"보다 `사람이 봤을 때 이해 가능한 직선·직각 회전 중심 패턴`을 더 우선하도록 바꾼 것입니다.

### 배경

기존 구조에서는 아래 문제가 반복됐습니다.

- 23번 토마토 옆 crop-side pose에서 다른 토마토를 누르면, 시작점을 같은 행 통로로 복귀하지 않고 목표 통로의 북/남 anchor처럼 먼 점으로 바로 잡는 경우가 있었음
- 같은 행 lateral transfer edge 때문에 그래프 최단경로는 짧아 보여도, 실제로는 사람이 기대하는 `직진 -> 90도 회전 -> 직진` 패턴과 어긋나는 경로가 나왔음
- 선택된 inspect waypoint에 거의 도착했는데도, executor가 다시 `final_target_pose`로 한 번 더 파고들면서 마지막 1m에서 좌우로 흔들림이 반복됐음
- 이 과정에서 `안전 통로 복귀를 재시도 중입니다` 상태가 목적지 근처에서도 다시 나타나 사용자가 보기엔 도착 실패처럼 느껴졌음

즉 문제의 본질은 단순히 로컬 컨트롤러 튜닝 부족이 아니라,
`시작 waypoint 선택`, `통로 갈아타기 방식`, `최종 목표 해석`이 서로 다른 철학을 섞어 쓰고 있었다는 점이었습니다.

### 주요 변경 사항

1. `manual_navigation_routing.py`에서 정책형 통로 경로 함수를 추가했습니다.

- `build_policy_waypoint_path(...)` 추가
- 같은 route 안에서는 기존 통로를 따라 이동
- 다른 route로 갈 때는 north/south endpoint를 계산해서 두 connector 후보를 비교한 뒤 더 짧은 쪽만 사용
- `build_manual_navigation_route(...)`, `estimate_navigation_route_cost(...)`, `_start_waypoint_route_cost(...)`가 모두 이 정책 경로를 사용하도록 변경

이제 경로 계산 단계부터 `직선 -> 직각 회전 -> 직선` 패턴이 더 강하게 보장됩니다.

2. 같은 행 inspect waypoint 사이의 lateral transfer를 기본적으로 끄도록 바꿨습니다.

- `ENABLE_INSPECT_LATERAL_TRANSFER_POLICY = False` 추가
- `build_waypoint_adjacency(...)`에서 lateral transfer edge를 기본적으로 넣지 않도록 변경

이 변경으로 이전처럼 `가까워 보여서 같은 행으로 옆 점프`하는 경로 대신,
정책적으로 정한 connector 기반 통로 갈아타기가 우선됩니다.

3. crop-side pose에서 시작할 때는 먼저 같은 행 egress inspect waypoint로 복귀하도록 고정했습니다.

- `select_start_waypoint_id(...)`에서 `select_route_egress_waypoint_id(...)`가 잡히면,
  더 먼 target lane anchor를 후보로 비교하지 않고 바로 그 egress를 시작점으로 반환하도록 변경

이렇게 해야 실제 로봇이 아직 통로 위에 있지 않은데도,
코드만 먼저 "이미 다른 통로 anchor에 올라탔다"고 가정해 버리는 문제를 막을 수 있습니다.

4. route 끝에 raw target pose를 무조건 덧붙이던 구조를 줄였습니다.

- `build_manual_navigation_route(...)`에서 explicit waypoint가 있는 명령은 마지막 목표를 해당 waypoint pose로 고정
- waypoint 그래프 밖 raw pose는 자동으로 덧붙이지 않도록 변경

이 변경은 마지막 짧은 비스듬한 세그먼트를 없애는 데 중요합니다.

5. executor에서 식물 이동의 실제 최종 목표를 `선택된 inspect waypoint`로 고정했습니다.

- `_configure_navigation_targets(...)`에서
  - `route_target_pose = selected_waypoint_pose`
  - `final_target_pose = selected_waypoint_pose`
  로 맞춤
- 기존 `candidate.final_target_pose`는 후보 설명용 데이터로만 남겨 두고,
  실제 실행은 선택된 정규 접근 waypoint 기준으로 처리

이제 식물 이동은 "통로 주행 -> 선택된 접근 waypoint 도착 -> 성공"으로 정리되며,
waypoint 도착 후 다시 raw crop-side pose로 한 번 더 파고들며 흔들리는 흐름이 크게 줄어듭니다.

6. 테스트를 새 정책 기준으로 다시 고정했습니다.

- home에서 lane_02로 갈 때는 `center inspect`를 억지로 거치지 않고, `lane_02_south_turn`에서 바로 올라가는 경로가 맞는 것으로 수정
- 23번 근처 crop-side pose에서 좌/우 lane 식물로 갈 때는 `같은 행 횡이동`이 아니라 `north connector 정책`을 타는 것으로 기대값 수정
- between beds에서 lane_02로 갈 때도 중심 통로를 따라 north connector를 거쳐 lane_02로 들어가는 정책 경로를 기대하도록 수정

### 기대 효과

- 같은 통로 안 이동은 더 일관되게 직진 위주로 보입니다.

## [S14P-짧은접근] [ROS/Nav] inspect waypoint 뒤에 짧은 직각 최종 접근 단계를 추가해 토마토 쪽 한 번 더 들어가게 조정

### 개요

이번 작업은 바로 앞선 정책형 통로 주행 리팩토링 위에, `inspect waypoint에 도착하자마자 바로 성공 처리되는 문제`를 보완하기 위해 진행했습니다.

핵심은 다음 한 줄로 정리할 수 있습니다.

- 장거리 이동은 기존 정책형 통로 주행을 그대로 유지
- inspect waypoint에 도착한 뒤에는 조건이 맞을 때만 `한 번 회전 + 짧은 직진`으로 토마토 쪽 최종 접근을 추가
- 이 마지막 접근이 실패해도 다시 전체 경로를 흔들리게 만들지 않고, 1회 재시도 후 안정성 우선으로 종료

즉, 이전처럼 토마토 앞에서 곡선 접근을 오래 끌지는 않되,
아예 inspect waypoint에서 끝나 버리지도 않도록 `짧고 제한된 마지막 접근 단계`를 별도로 넣었습니다.

### 배경

이전 리팩토링에서는 `선택된 inspect waypoint 자체를 최종 목표`로 두어,
raw crop-side pose로 다시 파고들며 흔들리던 문제를 줄였습니다.

하지만 그 결과,

- 직진 후 토마토 쪽으로 90도 꺾어도
- 마지막으로 토마토 가까이 한 번 더 들어가지 않고
- inspect waypoint 도달만으로 success가 떠 버리는 케이스가 생겼습니다.

반대로 예전에 raw final pose 전체를 그대로 다시 목표로 열었을 때는,

- 좌우로 심하게 까딱거림
- 안전 통로 복귀 재시도
- goal succeed 지연

같은 회귀가 반복됐습니다.

그래서 이번에는 `짧은 최종 접근`만 따로 떼어,
"켜야 할 때만 켜고, 실패해도 작게 끝나는" 구조로 보강했습니다.

### 주요 변경 사항

1. `ManualNavigationPhase`에 `FINAL_SHORT_APPROACH` 단계를 추가했습니다.

- 위치: `control_state.py`
- 이제 수동 이동 phase는
  - `ROUTE_EGRESS`
  - `ROUTE_ANCHOR`
  - `FINAL_OBSERVATION`
  - `FINAL_SHORT_APPROACH`
  로 구분됩니다.

이렇게 분리해야 마지막 1m 접근만 따로 제어하면서,
장거리 통로 주행 로직은 그대로 유지할 수 있습니다.

2. `ActiveCommandContext`에 짧은 최종 접근 상태를 추가했습니다.

- `final_short_approach_pose`
- `final_short_approach_retry_count`

즉 inspect waypoint 이후 마지막으로 더 들어갈지, 들어간다면 어디까지 갈지,
그리고 그 시도를 몇 번 했는지를 command context에 따로 기록하게 했습니다.

3. 짧은 최종 접근 pose를 만드는 helper를 추가했습니다.

- `normalize_heading_delta(...)`
- `snap_heading_to_cardinal(...)`
- `build_final_short_approach_pose(...)`

이 helper는 아무 경우나 마지막 접근을 켜지 않고,
아래 조건을 만족할 때만 최종 접근 pose를 만듭니다.

- inspect waypoint와 raw final pose 사이 거리가 너무 멀지 않을 것
- 대각선 접근이 아니라 축 정렬 접근일 것
- 헤딩은 0/90/180/-90도 중 하나로 정리될 것

즉, 이번 마지막 접근은 `정말 한 번 회전하고 짧게 직진하면 끝낼 수 있는 경우`에서만 켜집니다.

4. `_configure_navigation_targets(...)`에서 짧은 최종 접근 pose를 함께 계산하도록 바꿨습니다.

- 여전히 `route_target_pose`와 `final_target_pose`는 선택된 inspect waypoint pose로 유지
- 대신 조건이 맞을 때만 `final_short_approach_pose`를 별도로 생성

이 구조가 중요한 이유는,
기존 정책형 통로 주행을 깨지 않으면서도
inspect waypoint 이후 한 번 더 들어갈 수 있는 여지만 추가하기 때문입니다.

5. `_dispatch_current_navigation_stage(...)`에 `FINAL_SHORT_APPROACH` 분기를 추가했습니다.

- 이 단계에서는 일반 Nav2 목표나 긴 경로 계획을 다시 열지 않음
- 현재 pose에서 토마토 쪽으로
  - 회전 1회
  - 짧은 직진 1회
  만 실행

즉, 마지막 접근은 더 이상 전체 planner를 다시 부르는 단계가 아니라,
짧고 통제된 행동 시퀀스가 됩니다.

6. `_dispatch_final_short_approach_goal(...)`를 새로 추가했습니다.

이 함수는 다음 순서로 작동합니다.

1. 현재 pose를 읽음
2. 이미 충분히 가까우면 바로 success
3. 목표 쪽 heading을 cardinal 방향으로 정리
4. 필요하면 spin 1회
5. drive_on_heading 1회

즉 `제자리에서 한 번 방향 맞추고, 마지막으로 짧게 전진`하는 동작만 허용합니다.

7. 짧은 최종 접근 실패 시 fallback도 제한적으로 설계했습니다.

- 허용 오차 안이면 success
- 아니면 같은 짧은 접근 1회 재시도
- 그래도 안 되면 `정규 접근 waypoint까지는 안정적으로 도달했다`고 보고 종료

이 설계는 중요합니다.
예전처럼 마지막 접근 실패가 다시 전체 주행 실패나 egress 루프로 번지지 않게 하기 위함입니다.

### 기대 효과

- inspect waypoint에 도달한 뒤 토마토 방향으로 한 번 더 들어가는 동작이 생깁니다.
- 마지막 접근은 `직각 회전 + 짧은 직진` 패턴으로 제한돼, 긴 곡선 흔들림이 줄어듭니다.
- 마지막 1m 접근이 실패해도 전체 주행이 다시 꼬이지 않고, bounded retry 안에서 정리됩니다.
- 새로운 목적지로 다시 보낼 때도 장거리 이동 구조는 여전히 정책형 통로 주행을 유지합니다.

## [main-주석] [주행/상태동기화] inspect waypoint와 실제 작물 접근점을 다시 분리해 잘못된 성공 판정과 stale 상태 표시를 함께 정리

### 개요

이번 작업은 `22번 토마토 -> 21번 토마토` 같은 연속 이동에서 나타난 두 가지 문제를 함께 정리한 MR입니다.

1. 로봇이 실제 토마토 쪽으로 더 접근해야 하는데도, 통로 위 `inspect waypoint`까지만 간 뒤 `목적지에 도달했습니다.`로 성공 처리되던 문제
2. executor가 이미 `succeeded`를 기록했더라도, 시간이 조금 지나면 backend가 이를 `idle / 최신 status를 신뢰할 수 없음`으로 다시 덮어써 프론트엔드가 혼란스러워지던 문제

즉 이번 작업의 핵심은 **실제 최종 목표를 다시 올바르게 정의하고**, **이미 끝난 명령 상태는 stale로 덮어쓰지 않도록** 만드는 것입니다.

---

### 배경

현재 구조에서 식물 이동은 크게 두 점을 다룹니다.

- `route_target_pose`
  - 통로 위 관측 waypoint
- `final_target_pose`
  - 실제로 마지막에 더 접근해야 하는 작물 앞 목표점

그런데 최근 로직 변경 과정에서 이 둘이 다시 섞이면서,

- `final_target_pose`가 사실상 `inspect waypoint`와 같은 값으로 덮여 버렸고
- executor는 통로 waypoint까지만 도달해도 `succeeded`를 써 버렸습니다.

그래서 프론트에서는

- 로봇이 토마토 앞까지 가지 않았는데도 성공처럼 보이고
- 시간이 조금 지나면 또 stale 처리되어 `idle`로 보이는

상태 불일치가 생겼습니다.

---

### 주요 변경 사항

1. inspect waypoint와 실제 최종 접근점을 다시 분리하는 helper를 추가했습니다.

- 파일: `agribot_ws/src/agribot_bringup/agribot_bringup/robot_manual_command_executor.py`
- 추가 함수:
  - `resolve_observation_target_poses(...)`

이 함수는 다음 3가지를 한 번에 계산합니다.

- `route_target_pose`
  - 통로 위 inspect waypoint
- `final_target_pose`
  - 실제 토마토 앞 접근점
- `final_short_approach_pose`
  - 통로 waypoint에서 토마토 쪽으로 짧게 한 번 더 들어갈 수 있을 때만 쓰는 마지막 접근 목표

이렇게 분리해 두면, 앞으로도 “통로로 가는 단계”와 “토마토에 더 다가가는 단계”를 섞지 않고 유지할 수 있습니다.

2. executor가 더 이상 inspect waypoint를 최종 성공 목표로 오해하지 않도록 바꿨습니다.

- 파일: `agribot_ws/src/agribot_bringup/agribot_bringup/robot_manual_command_executor.py`

기존 문제:

- `context.final_target_pose`가 `selected_waypoint_pose`로 덮여서
- `-4.0, 6.0` 같은 통로 waypoint 도착만으로 성공 처리될 수 있었습니다.

수정 후:

- `route_target_pose`는 통로 waypoint로 유지
- `final_target_pose`는 다시 `selected_candidate.final_target_pose`를 유지

즉 이제는 executor가 “통로 도착”과 “실제 최종 접근”을 서로 다른 단계로 봅니다.

3. 짧은 최종 접근은 `route_anchor` 성공 직후에만 진입하도록 정리했습니다.

- 파일: `agribot_ws/src/agribot_bringup/agribot_bringup/robot_manual_command_executor.py`

이전에는 `FINAL_OBSERVATION` 성공 후 다시 `FINAL_SHORT_APPROACH`가 이어질 수 있는 구조가 일부 남아 있었습니다.
이번에는 흐름을 더 단순하게 정리했습니다.

- `ROUTE_ANCHOR` 도착
- 짧은 축 정렬 접근이 가능하면 바로 `FINAL_SHORT_APPROACH`
- 아니면 `FINAL_OBSERVATION`으로 일반적인 최종 접근

이렇게 해야 마지막 접근이 두 번 겹치거나, 이미 접근이 끝났는데 또 한 번 더 들어가려는 회귀를 줄일 수 있습니다.

4. `FINAL_SHORT_APPROACH` 실패를 더 이상 성공처럼 포장하지 않게 바꿨습니다.

- 파일: `agribot_ws/src/agribot_bringup/agribot_bringup/robot_manual_command_executor.py`

기존에는 짧은 최종 접근이 실패해도

- 재시도 후
- 필요하면 `정규 접근 waypoint까지는 왔다`는 이유로
- 사실상 성공처럼 끝날 수 있었습니다.

이제는 다음 순서로 처리합니다.

- 충분히 가까우면 성공
- 아니면 1회 재시도
- 그래도 안 되면 다른 observation candidate가 있으면 갈아탐
- 그것도 없으면 `failed`

즉 “실제로 접근 못 했는데 성공”이라는 잘못된 상태를 막았습니다.

5. backend는 이미 끝난 명령 상태를 stale로 덮어쓰지 않게 바꿨습니다.

- 파일: `backend/robot_runtime_state_service.py`

기존에는 status 파일이 오래되면, 그 안에 `succeeded`가 들어 있어도

- `executor가 최신 command status를 갱신하지 않아 현재 명령 실행 상태를 신뢰할 수 없습니다.`

로 다시 바뀔 수 있었습니다.

이번에는 `TERMINAL_COMMAND_STATUSES = {succeeded, failed, canceled}`를 두고,

- `pending/running` 같은 진행 중 상태만 stale 처리
- 이미 끝난 상태는 그대로 유지

하도록 바꿨습니다.

즉 프론트에서 “이미 도착했다고 떴다가 잠시 후 idle/stale로 바뀌는” 혼란을 줄였습니다.

6. 위 규칙을 테스트로 고정했습니다.

- 파일: `agribot_ws/src/agribot_bringup/test/test_robot_manual_command_executor.py`

- 파일: `backend/tests/test_robot_control_api.py`
  - `succeeded` 상태는 status 파일이 오래되어도 stale로 덮어쓰지 않는지 테스트 추가

## [S14P-실행복구] [ROS] 프론트 클릭 직후 executor가 죽어 로봇이 아예 안 움직이던 크래시를 복구

### 개요

프론트엔드에서 작물을 클릭하면 명령 파일은 정상적으로 기록됐지만, Gazebo에서 로봇이 전혀 움직이지 않는 문제가 있었습니다.

직접 로그를 확인해보니 주행 알고리즘 문제 이전에 `robot_manual_command_executor`가 명령을 받는 순간 바로 죽고 있었습니다.

이번 작업은 이 치명적인 실행 중단 원인을 먼저 복구해서, **최소한 프론트 클릭 후 명령이 executor 단계에서 바로 크래시로 사라지지 않도록** 만드는 데 집중했습니다.

---

### 배경

실제 런타임 상태를 확인한 결과는 다음과 같았습니다.

- 프론트엔드 `5173`, 백엔드 `8000`은 살아 있었음
- `/tmp/agribot_runtime/robot_manual_command.json`에는 최신 클릭 명령이 정상 기록됨
- 그런데 `/tmp/agribot_runtime/robot_manual_command_status.json`에는
  - `이전 executor 세션이 명령 완료 전에 종료되었습니다.`
  - `error=executor_restart`
  상태가 남아 있었음

이 상태에서 ROS 로그를 확인하니 원인은 더 분명했습니다.

- `robot_manual_command_executor`가 `_configure_navigation_targets(...)`에서
  `self._final_short_approach_enabled`
  를 읽는 순간 `AttributeError`로 종료됨
- 즉 프론트가 명령을 못 보내는 것이 아니라,
  **executor가 명령을 받는 즉시 예외로 죽어서 실제 주행까지 못 가는 상태**였습니다.

---

### 주요 변경 사항

1. `final_short_approach` 관련 파라미터를 실제 멤버 변수로 초기화하도록 복구했습니다.

- 파일: `agribot_ws/src/agribot_bringup/agribot_bringup/robot_manual_command_executor.py`

복구한 멤버는 다음과 같습니다.

- `_final_short_approach_enabled`
- `_final_short_approach_min_distance_m`
- `_final_short_approach_max_distance_m`
- `_final_short_approach_axis_tolerance_m`
- `_final_short_approach_success_xy_tolerance_m`
- `_final_short_approach_drive_speed_mps`
- `_final_short_approach_retry_limit`

이번 문제는 파라미터 `declare_parameter(...)`는 남아 있었지만,
실제로 `self._...` 멤버로 옮기는 구간이 빠져 있어 발생했습니다.

즉 선언은 있는데 실행 시 사용할 실제 값이 없었고,
명령이 처음 들어오는 순간 그 속성을 읽다가 바로 죽었습니다.

2. 왜 이 초기화가 중요한지 코드 주석도 보강했습니다.

이번 복구는 단순히 한 줄 추가가 아니라,
왜 이런 종류의 파라미터는 선언과 초기화가 항상 같이 있어야 하는지를
코드 안에서 바로 이해할 수 있게 설명을 덧붙였습니다.

---

### 기대 효과

- 프론트 클릭 후 executor가 즉시 `AttributeError`로 죽는 현상이 사라집니다.
- 명령 파일은 기록됐는데 로봇은 전혀 안 움직이던 직접 원인이 제거됩니다.
- 이후에 주행이 이상하더라도, 적어도 이제는 `주행 알고리즘` 자체를 실제로 검증할 수 있는 상태가 됩니다.

즉 이번 작업은 “주행 품질 개선”이 아니라, **주행 검증 자체가 가능하도록 기본 실행 경로를 다시 살린 작업**입니다.

---

### 검증

아래 검증을 수행했습니다.

- `python3 -m compileall agribot_ws/src/agribot_bringup/agribot_bringup/robot_manual_command_executor.py`
  - 통과
- `colcon build --packages-select agribot_bringup`
  - 통과
- `_final_short_approach_*` 멤버가 실제 초기화 구간에 존재하는지 코드 검색으로 확인

로그 기준으로도 이전에는 같은 위치에서 반복 크래시했는데,
이번 수정으로 적어도 그 직접 원인 자체는 코드상 제거됐습니다.

## [S14P-운영안정화] [ROS/Backend] 프론트 클릭 후 이동이 멈춘 것처럼 보이던 런타임을 자동 복구·자동 감지하도록 정리

### 개요

프론트엔드에서 목적지를 클릭하면 명령 파일은 실제로 써지는데, Gazebo/Nav2 런타임이 중간에 내려가면 화면에는 계속 `pending`처럼 보이거나 예전 상태가 남아 있던 문제가 있었습니다.

이번 작업은 주행 알고리즘 자체를 더 만지는 대신, **팀원이 같은 코드를 pull 받아도 기본 launch만으로 런타임이 더 잘 살아 있고, 죽었을 때는 프론트가 즉시 이상 상태를 보여주도록** 정리한 작업입니다.

핵심은 다음 4가지입니다.

1. `robot_manual_command_executor`에 heartbeat를 넣어 생존 여부를 주기적으로 기록
2. `simulation.launch.py`에서 runtime support 노드와 executor를 기본 respawn으로 변경
3. 시뮬레이션 기본값에서 공격적인 pose reset recovery를 꺼서 AMCL/TF/costmap 꼬임을 줄임
4. backend가 stale runtime과 `최신 명령 미반영` 상태를 즉시 감지해 프론트에 알려주도록 보강

---

### 배경

직접 확인해보니 문제의 직접 원인은 `프론트 클릭이 안 되는 것`이 아니었습니다.

- 프론트는 실제로 `POST /robot/commands`를 보내고 있었음
- `/tmp/agribot_runtime/robot_manual_command.json`에도 최신 명령이 기록됨
- 하지만 Nav2/Gazebo 런타임이 중간에 종료된 뒤
  - `robot_manual_command_status.json`
  - `robot_control_state.json`
  - `robot_navigation_path_snapshot.json`
  이 마지막 값만 남아 있었음
- 그 결과 프론트는 “명령을 보냈지만 왜 안 움직이지?”를 명확히 설명하지 못하고, 예전 `pending/running`처럼 보일 수 있었음

또 로그를 보면 planner 쪽에 `Robot is out of bounds of the costmap!`가 반복된 뒤 Nav2가 함께 종료되는 경우가 있었습니다.
이 상태에서 executor가 pose reset recovery를 강하게 다시 걸면, 오히려 TF/AMCL/costmap이 더 흔들릴 여지가 있었습니다.

즉 이번 문제는 “클릭 이벤트”보다 **런타임이 죽었을 때 기본 복구와 상태 노출이 약했던 구조 문제**로 보는 것이 맞았습니다.

---

### 주요 변경 사항

1. `robot_manual_command_executor`에 control state heartbeat를 추가했습니다.

- 파일: `agribot_ws/src/agribot_bringup/agribot_bringup/robot_manual_command_executor.py`
- 새 파라미터:
  - `control_state_heartbeat_period_sec`
- 기본 heartbeat:
  - `1.0초`

이제 executor는 상태 변화가 없어도 `robot_control_state.json`을 주기적으로 다시 기록합니다.
이 heartbeat는 “지금 executor가 실제로 살아 있는가”를 프론트/백엔드가 판단하는 기준이 됩니다.

즉,

- 예전: 마지막 pending 값이 남아 있으면 계속 살아 있는 것처럼 보일 수 있었음
- 지금: heartbeat가 멈추면 backend가 stale runtime으로 판정

2. `simulation.launch.py`에서 runtime support 노드를 기본 respawn으로 바꿨습니다.

- 파일: `agribot_ws/src/agribot_bringup/launch/simulation.launch.py`
- 대상 노드:
  - `runtime_snapshot_exporter`
  - `robot_manual_command_executor`
  - `mission_bridge_executor`
- 공통 설정:
  - `respawn=True`
  - `respawn_delay=2.0`

즉 이 노드들이 예기치 않게 죽어도, 팀원이 별도 수동 조작을 하지 않아도 launch 기본값만으로 자동 재기동됩니다.

3. 시뮬레이션 기본 launch에서 공격적인 pose reset recovery를 껐습니다.

- 파일: `agribot_ws/src/agribot_bringup/launch/simulation.launch.py`
- 변경:
  - `simulation_pose_reset_recovery_enabled: False`

이 설정을 기본으로 둔 이유는, 지금 프로젝트에서는 Gazebo pose warp와 `/initialpose` 재주입이 한 번 꼬이면
오히려 `AMCL -> TF -> costmap -> planner` 전체를 더 불안정하게 만들 수 있었기 때문입니다.

즉 “죽으면 억지로 위치를 다시 밀어넣어 살린다”보다, **기본 launch는 더 보수적으로 유지하고 respawn + stale 감지로 대응**하도록 방향을 정했습니다.

4. Nav2 쪽도 기본 respawn을 켰습니다.

- 파일: `agribot_ws/src/agribot_navigation/launch/navigation.launch.py`
- 변경:
  - `use_respawn` 기본값 `false -> true`

즉 팀원이 특별히 옵션을 바꾸지 않아도 Nav2 노드가 기본적으로 자동 복구 방향을 타게 됩니다.

5. backend에 stale runtime 감지와 최신 명령 미반영 감지를 넣었습니다.

- 파일: `backend/robot_runtime_state_service.py`

추가된 기준은 두 가지입니다.

- `robot_control_state.json`이 너무 오래 갱신되지 않으면
  - `executor heartbeat가 오래 멈춰 있어 현재 ROS 런타임이 비정상으로 보입니다.`
- `robot_manual_command.json`이 더 최근인데 `robot_manual_command_status.json`이 예전 command id에 머물면
  - `최신 이동 명령은 기록됐지만 executor가 아직 이 명령을 처리하지 못했습니다.`

즉 이제 프론트는 단순히 `pending`만 보는 게 아니라,

- 런타임이 멈춘 상태인지
- 최신 명령이 executor에 반영되지 못한 상태인지

를 더 정확하게 알 수 있습니다.

6. 이 새 판단 기준을 backend 테스트로 고정했습니다.

- 파일: `backend/tests/test_robot_control_api.py`

추가 테스트:

- control state heartbeat가 멈춘 파일은 stale runtime으로 판정되는지
- status 파일보다 command 파일이 더 최신이면 “최신 명령 미반영” 상태로 노출되는지

즉 이번 변경은 눈으로 한 번 확인하는 수준이 아니라, 회귀 테스트로 남겼습니다.

---

### 기대 효과

- 프론트 클릭 자체는 되는데 로봇이 안 움직이던 상황에서, 이제 “왜 안 움직이는지”가 더 명확하게 드러납니다.
- executor/runtime support/Nav2가 죽어도 기본 launch 설정만으로 자동 재기동이 걸립니다.
- 마지막 pending 상태가 남아 사용자에게 오해를 주는 문제를 줄입니다.
- 팀원이 같은 코드를 pull 받아도 별도 개인 설정 없이 같은 자동 복구 기본값을 사용합니다.

## 핵심 확인 로그 요약

- `ros2 topic list`에서 다음 핵심 토픽 확인:
  - `/agribot/camera/image`
  - `/agribot/camera/depth_image`
  - `/agribot/camera/camera_info`
  - `/agribot/imu`
  - `/agribot/lidar/scan`
  - `/odom`
- `/agribot/camera/image`, `/agribot/camera/depth_image`, `/agribot/camera/camera_info`, `/agribot/imu`, `/odom`은 `ros2 topic echo --once`로 메시지 수신 확인
- `/agribot/lidar/scan`은 `ros2 topic echo --once`와 `gz topic -e -n 1` 모두 타임아웃
- `/clock` 브리지를 추가한 뒤 `ros2 topic echo --once /clock` 수신 확인

## bridge 수정 포인트

1. Gazebo를 기본 재생 상태로 실행해야 함
   - 기존 런치는 월드만 전달해서 Gazebo가 정지 상태로 시작했고, 이 경우 토픽 이름만 보이고 메시지는 흐르지 않았음
   - `spawn_agribot.launch.py`에서 `gz_args`에 `-r`을 추가해 자동 재생으로 변경

2. 단일 `parameter_bridge`에 카메라 토픽을 모두 묶으면 카메라 메시지 전달이 불안정했음
   - 기존 구성에서는 `/agribot/camera/image`, `/agribot/camera/depth_image`, `/agribot/camera/camera_info`가 ROS 측에서 수신되지 않았음
   - 분리 검증 결과:
     - `ros_gz_image image_bridge /agribot/camera/image` 사용 시 RGB 이미지 수신 성공
     - `ros_gz_image image_bridge /agribot/camera/depth_image` 사용 시 Depth 이미지 수신 성공
     - 별도 `parameter_bridge`로 `/agribot/camera/camera_info`만 연결 시 수신 성공
   - 따라서 메인 런치도 이미지 전용 브리지와 개별 `parameter_bridge`로 분리

3. `/clock` 브리지가 필요함
   - 이후 Nav2, RViz2, bag 기록, `use_sim_time` 설정을 위해 `/clock`이 필요
   - 메인 상태 브리지에 `/clock@rosgraph_msgs/msg/Clock[gz.msgs.Clock` 추가

4. LiDAR는 bridge보다 센서 발행 자체를 먼저 의심해야 함
   - `/agribot/lidar/scan`은 별도 브리지로도 미수신
   - 현재 단계에서는 `gpu_lidar` 센서 또는 렌더링/물리 부하 쪽 이슈 가능성이 큼

## 추가 관찰 사항

### 1. 실시간 성능 저하

- `ros2 topic hz /agribot/imu` 결과:
  - 평균 약 `0.635 Hz`
- 반면 GZ 헤더 기준 IMU는 `100 Hz`로 정상
- 즉, 센서 설정 자체보다는 Gazebo 실시간 성능이 심하게 떨어지고 있음

### 2. 물리 충돌 병목 후보

- Gazebo 실행 중 `ODE Message 2: Trimesh-trimesh contact hash table bucket overflow` 경고가 반복 발생
- 다음 모델들이 모두 시각 메쉬를 그대로 충돌 메쉬로 사용 중:
  - `agribot_ws/src/agribot_description/models/greenhouse/model.sdf`
  - `agribot_ws/src/agribot_description/models/tomato_plant/model.sdf`
  - `agribot_ws/src/agribot_description/models/tomato/model.sdf`
- 월드에는 위 메쉬 기반 모델이 다수 포함되어 있어 물리 계산 부하가 큼

## 후속 작업 제안

1. `S14P-32` 전에 LiDAR 미발행 원인을 먼저 분리
   - `gpu_lidar`를 `lidar` 계열 센서로 바꾸는 비교 테스트
   - 헤드리스(`-s`) 실행 시 LiDAR 발행 여부 비교

2. 물리 성능 개선
   - 온실/식물/토마토 충돌체를 box, cylinder, sphere 등 단순 충돌체로 교체
   - 필요한 경우 토마토를 `static`으로 바꾸거나 충돌 비활성화 검토

3. frame 정합 준비
   - 현재 수신된 토픽 기준으로 `camera_link`, `imu_link`, `base_link`, `odom` 프레임 관계를 다음 티켓에서 이어서 검증
